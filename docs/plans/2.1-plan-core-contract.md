# 2.1 Plan Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build the inspection core on top of the shared 0.1 contract: tag config validation, event model, rule engine, state transitions, persistence, CSV export, and regression tests.

## Task 2.1.1: 标签配置校验

**Files:**

- Modify: `src/inspection_robot/config.py`
- Modify: `config/tag_map.json`
- Create: `tests/test_config.py`

**Step 1: 扩展配置字段**

`config/tag_map.json` 支持：

```json
{
  "1": {
    "name": "Apple",
    "zone": "A区",
    "expected_zone": "A区",
    "priority": 1,
    "kind": "item"
  }
}
```

`kind` 允许：

- `item`：物品标签。
- `zone`：分区站点标签。

**Step 2: 保持旧配置兼容**

如果旧配置没有 `priority`，默认 `1`。

如果旧配置没有 `kind`，默认 `item`。

**Step 3: 增加校验**

`load_tag_map()` 对以下情况抛出 `ValueError`：

- 缺少 `name`
- 缺少 `zone`
- 缺少 `expected_zone`
- `priority` 不是正整数
- `kind` 不是 `item` 或 `zone`

**Step 4: 写测试**

`tests/test_config.py` 覆盖：

- 当前 `config/tag_map.json` 可加载。
- 旧格式可补默认值。
- 非法配置抛出清晰错误。

**Step 5: Verify**

```bash
py -3 -m unittest tests.test_config -v
```

Expected: 所有配置测试通过。

## Task 2.1.2: 事件模型与规则函数

**Files:**

- Create: `src/inspection_robot/core/__init__.py`
- Create: `src/inspection_robot/core/events.py`
- Create: `src/inspection_robot/core/rules.py`
- Create: `tests/test_rules.py`

**Step 1: 建立事件创建函数**

在 `events.py` 提供：

```python
def make_event(
    event_type: str,
    tag_id: str | None,
    item: str,
    zone: str,
    expected_zone: str,
    priority: int,
    status: str,
    message: str,
) -> dict[str, object]:
    ...
```

要求：

- `id` 使用稳定前缀，例如 `evt-`。
- `time` 使用 ISO 格式。
- 字段与 `docs/api_contract.md` 一致。

**Step 2: 实现基础规则**

在 `rules.py` 实现：

- 正常标签：`normal_tag`
- 未知标签：`unknown_tag`
- 错放标签：`wrong_zone`

**Step 3: 实现重复规则**

同一轮巡检中，同一个 item tag 如果出现在不同分区，产生 `duplicate_tag`。

**Step 4: 实现缺失规则**

在分区结束或整轮结束时，某分区应出现但没出现的 item tag 产生 `missing_tag`。

**Step 5: 写测试**

`tests/test_rules.py` 至少覆盖：

- 正常标签。
- 未知标签。
- 错放标签。
- 重复标签。
- 缺失标签。

**Step 6: Verify**

```bash
py -3 -m unittest tests.test_rules -v
```

Expected: 每种事件类型都有独立测试。

## Task 2.1.3: Store 与状态机重构

**Files:**

- Create: `src/inspection_robot/core/store.py`
- Modify: `src/inspection_robot/state.py`
- Create: `tests/test_store.py`

**Step 1: 保留当前公开行为**

这些现有流程不能断：

1. `POST /api/start`
2. `POST /api/simulate/tag/1`
3. `POST /api/simulate/tag/4`
4. `POST /api/confirm`
5. `GET /api/export.csv`

**Step 2: 增加 robot 输入方法**

`InspectionStore` 支持：

```python
record_tag(tag_id, observed_zone=None, source="simulate")
record_obstacle(distance_mm, blocked)
record_robot_status(status, message=None)
finish_run()
```

**Step 3: 状态转换统一**

至少实现：

- `start()` -> `PATROL`
- 正常标签 -> `NORMAL_LOGGED`
- 异常标签 -> `WAIT_CONFIRM`
- `confirm()` -> `CONFIRMED` then `PATROL`
- 障碍 blocked -> `OBSTACLE_WAIT`
- 障碍 clear -> `PATROL`
- `stop()` -> `STOPPED`

**Step 4: 写 store 测试**

`tests/test_store.py` 覆盖：

- 正常标签写入事件。
- 错放标签变为 pending。
- confirm 只关闭 pending 事件。
- 障碍状态进入和解除。
- 旧事件不被覆盖。

**Step 5: Verify**

```bash
py -3 -m unittest tests.test_store -v
```

Expected: 所有状态转换确定可测。
