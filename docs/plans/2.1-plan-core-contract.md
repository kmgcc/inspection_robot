# 2.1 队友计划：固定地图、路径规划与货架异常规则

**负责人：** 队友 2  
**前置依赖：** `docs/api_contract.md`、`config/warehouse_map.json`、`config/shelf_manifest.json`、`config/tag_map.json`  
**目标：** 在不依赖真实小车硬件的情况下，实现固定仓库地图、禁区规避路径规划、货架扫描状态机、多模态物品识别证据合并、物品清单比对、事件持久化和测试。

本计划是系统的“脑子”，但不是电机驱动。它只接收 1.1 上报的观察，决定当前目标、路径状态和异常事件；它不直接打开摄像头、不直接读超声波、不直接控制电机。观察可以来自 AprilTag、OCR、颜色、图像分类、超声波或四路红外黑胶带传感器。这样可以在本地用模拟数据先跑通，等 1.1 真车输入完成后再合并。

**Step 1：加载仓库地图**

定义 `load_warehouse_map(root)`，读取：

```json
{
  "grid_size": [8, 6],
  "start": [0, 0],
  "home": [0, 0],
  "forbidden_cells": [[2, 2], [2, 3]],
```

- `grid_size` 是两个正整数。

- 禁区格在地图内。
- 每个货架有 `scan_pose` 和 `safe_side`。

**Step 2：加载货架清单**

定义 `load_shelf_manifest(root)`。每个货架必须有 `expected_items`，并且是非空字符串列表。

**Step 3：扩展标签字典**

`tag_map.json` 支持：

```json
{
  "118": {
    "name": "A1",
    "kind": "shelf",
    "shelf_id": "A1",
    "marker_family": "TAG36H11",
    "ocr_label": "A1"
  },
  "46": {
    "name": "Red Bottle",
    "kind": "item",
    "item_id": "item_46",
    "expected_shelf": "A1",
    "marker_family": "TAG36H11",
    "expected_color": "RED",
    "expected_ocr": "ITEM-46",
    "expected_image_class": "BOTTLE",
    "priority": 1
  }
}
```

`kind` 允许 `shelf`、`item`、`zone`、`pose`。旧字段 `zone`、`expected_zone` 继续兼容，但新版规则优先使用 `shelf_id` 和 `expected_shelf`。配置校验必须检查 AprilTag ID 范围：物品 `1-50`，货架 `101-120`，定位点预留 `201-220`，特殊区域预留 `301-320`。物品条目建议包含 `expected_color`、`expected_ocr`、`expected_image_class` 中的至少一项；货架条目建议包含 `ocr_label`。

## Task 2.1.3：货架扫描规则

**Files:**

- Create: `src/inspection_robot/core/rules.py`
- Create: `src/inspection_robot/core/events.py`
- Create: `tests/test_shelf_rules.py`

**Step 1：事件创建函数**

`events.py` 提供：

```python
def make_event(
    event_type: str,
    item: str = "-",
    tag_id: str | None = None,
    shelf_id: str | None = None,
    expected_shelf: str | None = None,
    priority: int = 1,
    status: str = "info",
    message: str = "",
    source: str = "core",
) -> dict[str, object]:
    ...
```

基础字段必须兼容 `docs/api_contract.md`。

**Step 2：扫描结果比对**

`rules.py` 提供：

```python
def evaluate_shelf_scan(
    shelf_id: str,
    detected_items: list[str],
    manifest: dict[str, dict[str, list[str]]],
    tag_map: dict[str, dict[str, object]],
) -> list[dict[str, object]]:
    ...
```

规则：

- 应有但没识别到：`missing_item`
- 同一货架同一物品出现多次：`duplicate_item`
- 物品期望货架不是当前货架：`wrong_shelf`
- 标签不在字典：`unknown_item`
- AprilTag 主身份与 OCR、颜色或图像证据冲突：`evidence_mismatch`
- 完全匹配：每个物品可记 `normal_item`，或只记一条 `shelf_scanned`

建议增加一个更完整的入口，用于接收 1.1 的多模态观察：

```python
def evaluate_detection_evidence(
    shelf_id: str,
    detections: list[dict[str, object]],
    manifest: dict[str, dict[str, list[str]]],
    tag_map: dict[str, dict[str, object]],
) -> list[dict[str, object]]:
    ...
```

其中每个 detection 可包含 `tag_id`、`ocr_text`、`color`、`image_class`、`marker_family`、`confidence`。规则优先级为：AprilTag 数字 ID 确定主身份；OCR/颜色/图像和配置不一致时生成 `evidence_mismatch`，但不直接改写主身份。

**Step 3：测试场景**

至少覆盖：

- A1 完全正常。
- A1 缺少 `item_03`。
- A1 识别到两个 `item_02`。
- A1 识别到期望在 A2 的物品。
- A1 识别到未知标签。
- A1 中某个物品 AprilTag 正常，但颜色或图像类别与配置不一致。
- 货架 AprilTag 映射为 A1，但 OCR 读成 A2，生成复核事件或警告。

**Verify:**

```bash
py -3 -m unittest tests.test_shelf_rules -v
```

Expected: 每种异常都有独立测试。

**Step 2：增加新版输入方法**

```python
record_pose(x, y, heading, source="runtime")
record_path(waypoints, status="active")

record_scan_result(shelf_id, detected_items, frame_id=None)

```

旧的 `record_tag()` 继续保留，可在过渡期把标签识别转成扫描结果。

**Step 3：状态转换**

至少实现：

- `start()` -> `PLANNING`
- 路径生成成功 -> `PLAN_READY`
- 开始执行 -> `MOVING`
- 到达货架点 -> `ALIGNING_SHELF`
- 开始扫描 -> `SCANNING_SHELF`
- 扫描完成 -> `ANALYZING`
- 正常 -> `NORMAL_LOGGED` then `MOVING`
- 异常 -> `WAIT_CONFIRM`
- 障碍 -> `OBSTACLE_WAIT`
- 重规划 -> `REROUTING`
- 黑胶带禁区触发 -> `FORBIDDEN_ZONE_WAIT`
- 完成 -> `FINISHED`
- 停止 -> `STOPPED`

**Step 4：`/api/status` 快照**

`snapshot()` 返回旧字段和新版扩展字段。旧看板可继续显示，3.1 新看板可直接读取 `current_shelf`、`path`、`shelves`、`scan`。
