巡检小车五项问题修复计划

 背景（为什么做这次改动）

 RASPBOT V2 巡检小车目前存在五个现场可见的问题，且用户新增一条视觉架构约束。核心症结集中在运动控制的多线程竞态、边界检
 测只看单帧瞬时快照、巡逻速度过快、OCR/识别流程从未真正接入主线、以及缺少直线行驶的姿态闭环。本计划一次性修复，并把"检
 测到物体减速/停车"改成由高速粗检测触发、识别（OCR/AprilTag）只在停稳后执行的分层架构。

 关键架构约束（用户新增，务必贯穿问题四）

 "检测到物体→减速/停车→识别" 的触发不能依赖二维码(AprilTag)或文字(OCR)。

 - 触发 = 高速粗检测：用一个"只判断画面里有没有物体/方块轮廓"的快速、廉价、可高帧率检测（例如 YOLO，或轻量的 OpenCV
 轮廓/方框检测）。它只回答"有没有目标"，不回答"是什么"。这样小车在行驶过程中就能实时发现目标并减速/停下。
 - 识别 = 停稳后再做：只有在停车（画面稳定、无运动模糊）之后，才运行重的识别——AprilTag(tag36h11) + PaddleOCR 中文 +
 图像分类 + 颜色。有 tag 以 tag 为主身份；无 tag 时允许"形状+文字"形成保守观察。
 - 现状问题正好相反：现在只有先识别到 AprilTag 才会跑 OCR，没有 tag
 就完全不识别，且识别在行驶中做（会糊）。必须把"触发"与"识别"解耦。

 实现要点：
 - 新增快速粗检测 detect_object_presence(frame, cv2, roi) -> bool（vision/tag_detector.py）。默认用轻量 OpenCV
 轮廓/方框检测（复用现有 _classify_image_region 的思路但只返回布尔"存在"，阈值放宽、速度优先）。这一路不依赖
 AprilTag 或 OCR，是行驶中减速/停车的主触发。
 - 可选 YOLO 后端只作为增强层：当前小车未检测到 Hailo 运行时或 /dev/hailo*，因此不能把 AI HAT+ 加速当作默认可用；
 有 Hailo 驱动和 HEF 模型后再启用 OBJECT_DETECTOR=hailo_yolo。已有 /home/pi/YOLOv5-Lite-master 可作 CPU 备选，
 但不得替代 OpenCV 主触发。
 - 巡逻主循环每步抓一帧调用粗检测：命中→减速一档→停车→跑完整识别→记录证据+播报→继续。
 - 完整识别复用 _collect_detections()，但只在停稳状态触发（见问题四）。

 - 小车已有 /home/pi/yolov5-lite（Python 虚拟环境）和 /home/pi/YOLOv5-Lite-master（源码与权重）。虚拟环境已有
 torch 2.4.0、torchvision 0.19.0、opencv 4.10.0、numpy 1.23.5；未安装 paddleocr、paddlepaddle、onnxruntime。

 粗检测模型选择：
 1. **默认主方案：OpenCV 轮廓/方框粗检测。**
    - 用途：行驶中高频判断"画面 ROI 里是否有卡片/方块/货架槽位轮廓"。
    - 理由：不需要新依赖、不占大量磁盘、CPU 负担低、启动快；最符合"只触发减速/停车，不负责识别内容"的需求。
    - 目标频率：跟随巡逻步进做 10Hz 量级采样；连续 2-3 帧命中再减速/停车，避免单帧噪声。
 2. **CPU 备选：YOLOv5-Lite-e，使用现有 /home/pi/YOLOv5-Lite-master/weights/v5lite-e.pt。**
    - 用途：仅在 OpenCV 误触发太多、且暂时没有 Hailo 时，作为 1-2 FPS 的低频复核或离线对比工具。
    - 实测：在小车 Pi 5 CPU 上跑自带 bus.jpg，320 输入单张约 0.54s。能用，但不适合每帧行驶中触发。
    - 不推荐默认用 v5lite-s / v5lite-c：它们更重；当前任务只需要"有没有目标"，不需要更大 COCO 模型。
    - 不推荐直接用已有 best.pt / gabarge.pt 做巡检主模型：这些权重主要是交通/垃圾分类或课程样例，不是货架卡片/槽位/方块
      的专用检测；最多当作验证脚手架。
 3. **Hailo 加速推荐：AI HAT+ 13 TOPS 就绪后，优先用 Raspberry Pi 官方 rpicam Hailo YOLO demo 做冒烟测试，再接
    Hailo Model Zoo 的 HAILO8L nano/small 检测 HEF。**
    - 官方 Raspberry Pi 文档说明 AI HAT+ 13 TOPS 基于 Hailo-8L，可通过 rpicam-apps / Picamera2 对支持的视觉模型做
      NPU 推理；同时需要安装 hailo-all、rpicam-apps 和匹配版本模型。
    - 冒烟测试优先使用 /usr/share/rpi-camera-assets/hailo_yolov8_inference.json 或 hailo_yolov5_inference.json。
    - 若需要自训练少类别模型，优先选 Hailo 官方已支持的 nano 级检测模型（如 HAILO8L 列表里的 yolov11n/yolov10n 或
      Raspberry Pi 资产里的 YOLOv8），类别只保留 card / shelf_slot / box_or_object / unknown_object。
    - 不把 YOLOv5-Lite .pt 直接宣称为 Hailo 加速模型；它需要另行导出、量化、编译成 HEF，并和板上 HailoRT 版本匹配。

 OCR 模型选择：
 1. **推荐：PP-OCRv5_mobile_det + PP-OCRv5_mobile_rec。**
    - 用途：只在停车后，对透视矫正后的局部文字 ROI 做中文/英文/数字识别。
    - 理由：官方 PaddleOCR 文档中 mobile 检测模型约 4.7MB、mobile 识别模型约 16MB，比 server 模型更适合树莓派；
      PP-OCRv5 支持简体中文、繁体中文、英文、日文等文本类型。
    - 配置：关闭文档方向分类、整页矫正等文档 OCR 功能；必要时才开文字行方向分类。我们识别的是标签小块，不是整页文档。
 2. **不推荐：PP-OCRv5_server、PaddleOCR-VL、任何 VLM/LLM OCR。**
    - 理由：当前剩余磁盘只有约 7.3GB，OCR 又不是行驶中实时触发；大模型会增加安装、内存和启动风险，收益不匹配。
 3. **回退：pytesseract 只作英文/数字兜底。**
    - 当前代码的 pytesseract 路线会丢中文，且识别质量依赖系统语言包；不能作为中文主 OCR。

 关键原则：
 - 粗检测负责"该不该慢下来/停下"，不负责识别是谁。
 - AprilTag 和 OCR 负责"停稳后识别是谁/是否冲突"，不负责行驶中触发停车。
 - Hailo 只加速神经网络检测；AprilTag、OpenCV 轮廓、PaddleOCR CPU 推理、业务状态机仍由 Pi 5 负责。
 - 首版必须做到：无 Hailo 也能靠 OpenCV 粗检测 + 停车后 AprilTag/OCR 跑完整演示；Hailo 是后续高帧率增强，不是上线前置条件。

 参考官方资料：
 - Raspberry Pi AI HAT+ / AI software 文档：AI HAT+ 13 TOPS 是 Hailo-8L，适合物体检测等视觉 NPU 任务；需要安装 hailo-all
   和 rpicam-apps，官方提供 YOLOv5/YOLOv8/YOLOX 等 rpicam demo。
 - Hailo Model Zoo：Hailo-8L 模型需使用匹配的 Hailo 工具链和 HEF，不能直接拿 .pt 当作 NPU 模型。
 - PaddleOCR / PaddleX 文档：PP-OCRv5 mobile 检测和识别模型是边缘设备优先方案，server/VLM 类模型不适合本项目首版。

 问题一：网页右上角"立即停止"按钮无法真正立即停止

 根因

 - 多线程竞态、无锁：RobotRuntime.stop()（runtime.py:179）在 Flask 线程里调 motion.stop()，但巡逻后台守护线程（_run_con
 tinuous_patrol_safely）随后又会发出新的电机指令。motion._call_motion()（motion.py:60）没有任何停止标志感知，会照常
 func(speed) 并 time.sleep(duration) 睡满。
 - 长阻塞段不查停止标志：_settle() 睡 0.45s（runtime.py:1118）；避障前进 watch_boundary=False
 时走非分块分支睡满时长（runtime.py:603-608）；IMU 闭环转向 turn_90_with_gyro 多脉冲循环 + 每次
 settle_seconds（mpu6050.py:255,296）完全不看运行时停止标志。
 - 前端顺序等待：dashboard.js:630-645 先 await /api/test/stop（只停测试会话，与巡逻无关）再 await /api/stop。

 改动

 1. motion.py 增加全局急停闸（单一收口点，所有运动都经过这里）：
   - 模块级 _ABORT = threading.Event()；新增 request_stop()（置位 _ABORT +
 立即硬件停车）、clear_stop()、is_stop_requested()。
   - 新增 _interruptible_sleep(seconds)：以 ~0.01s 小片轮询 _ABORT，置位即返回。
   - _call_motion()：进入即判 _ABORT（置位则 stop 并 return，不再驱动）；每次 func() 前再判一次；两处 time.sleep 换成
 _interruptible_sleep；结束若 _ABORT 置位则再 stop 一次。→
 保证急停后后台线程不会再驱动电机，且任何正在进行的睡眠立即唤醒。
 2. runtime.py：
   - stop()：改调 motion.request_stop()（getattr 保护），同时 self._stop_event.set()。
   - start()：启动线程前 motion.clear_stop()，避免上一次急停残留把新巡逻卡死。
   - 新增 self._interruptible_sleep(seconds)（按 self._stop_event 分片），替换 _settle()、_confirm_boundary_state
 间隔、丢线等待等纯 time.sleep。
   - _turn_90() 把 should_abort=self._stop_event.is_set 传入 IMU 转向。
 3. mpu6050.py：turn_90_with_result / turn_90_with_gyro 增加可选 should_abort 回调；循环顶部与 settle_seconds
 睡眠处检查它，置位即尽快退出。（因 _call_motion 已急停感知，脉冲本身瞬停，这里只需让循环别再空转。）
 4. web.py：/api/stop 做成"全局急停"——同时 _stop_test_session() + runtime.stop() + motion.request_stop()（try/except
 RobotHardwareError）。/api/control（手动指令）里 runtime.stop() 之后、发指令之前必须
 motion.clear_stop()（否则手动指令会被急停闸吞掉）。
 5. test_mode.py：TestSessionManager.stop() 用 motion.request_stop()；_start_session() 先
 motion.clear_stop()（急停后仍能重新做测试）。
 6. dashboard.js：急停按钮改为并行触发（Promise.allSettled([postJson('/api/stop'),
 postJson('/api/test/stop')])），/api/stop 优先，去掉顺序等待延迟。

 关键文件：robot/motion.py、runtime.py、robot/mpu6050.py、web.py、test_mode.py、static/dashboard.js

 问题二：四路寻线到边界（黑带）漏检、斜过冲过

 根因

 - 边界判定只看单帧瞬时快照是否 4 路全黑（_candidate_boundary →
 full_tape_boundary_detected，runtime.py:430、sensors.py:87）。高速或斜着过带时，任何一帧都凑不齐"4 路同时为
 0"，于是漏检。
 - _pending_boundary_state 只锁存一次完整读数（runtime.py:628），不是"短时间窗内先后点亮的累积"。
 - 现有 boundary_confirm_samples 是"跨时间必须每次都黑"的 AND 滤波（runtime.py:438），方向相反、会让高速漏检更糟。

 改动（时间窗 OR 累积 = 用户说的"极短时间差内先后点亮也算同时"）

 1. RobotRuntimeConfig 新增 boundary_window_seconds（默认 0.15，env BOUNDARY_WINDOW_SECONDS）。
 2. 新增状态 self._black_seen_at = [-1e9]*4；新增 _feed_boundary_window(tape_state) -> bool：
   - 每次读数：对为 0（黑）的通道记录 now 时间戳。
   - 统计"最近 window 秒内曾为黑"的通道数 ≥ boundary_min_black_sensors 即判定到边界。
   - 单帧就 4 路全黑时自然立即命中（时间窗对它无副作用）。
 3. 接入两个采样点：
   - _poll_boundary_during_motion（运动中每 ~0.02–0.04s 采样，runtime.py:621）用 _feed_boundary_window 替换
 _candidate_boundary，命中即锁存并停车。
   - _handle_tape_boundary（runtime.py:326）主入口用窗口判定，保留 _pending_boundary_state 传递机制。
   - 移除/中性化 _confirm_boundary_state 的再确认（窗口已完成确认；否则锁存的部分状态会被瞬时 4 路判定误拒）。
 4. 处理完一次边界（_handle_planned_boundary_turn 转向后、_handle_forbidden_bypass
 绕行后）与巡逻启动时，_reset_boundary_window() 清空时间戳；配合已有 boundary_cooldown_seconds 防止立刻重复触发。
 5. 保持"少防抖"：boundary_confirm_samples 维持 1（等于不做时间 AND
 滤波）。因为正常地面永不触发（全白），窗口累积不会误触发。

 ▎ 与问题三/五协同：巡逻降速 + 陀螺仪保持直线，会显著降低斜着过带概率，进一步提高命中率。

 关键文件：runtime.py、（判定复用 robot/sensors.py 现有原语）

 问题四：PaddleOCR / "检测到物体停下识别" 完全无效果

 根因

 - 识别主循环默认关闭：scan_enabled 默认 False（runtime.py:67），连续巡逻明确"暂不做货架识别"。→ 识别根本没被调用。
 - 连续巡逻不先停车：_scan_visible_shelf（runtime.py:715）扫描前不 motion.stop()（不同于航点版 _scan_shelf）。
 - 线上 OCR 其实是 pytesseract 且被注释掉（requirements.txt:7），ImportError
 静默返回空；_clean_ocr_text（tag_detector.py:437）只保留 ASCII，中文被全删。PaddleOCR 只存在于离线评测脚本
 scripts/evaluate_ocr_engines.py，从未接入。
 - OCR/识别被 AprilTag 门控：_detect_frame（tag_detector.py:156）只在检测到 tag 的循环里跑 OCR/分类。无 tag →
 全程无识别。
 - 稳定性 tracker（min_stable_frames=3）在运动中难以攒够稳定帧，会抑制输出（停车后可满足）。

 改动（落实"粗检测触发 / 停稳后识别"的分层架构）

 1. 接入 PaddleOCR 中文（用户选定），tag_detector.py::_try_ocr_text：
   - 优先 PaddleOCR PP-OCRv5 mobile 组合：PP-OCRv5_mobile_det + PP-OCRv5_mobile_rec。模块级缓存实例，构造昂贵只建一次；
 ROI 转 RGB 传入，解析参考 evaluate_ocr_engines.py::_paddle_text；失败回退 pytesseract；都无则返回空。
   - OCR 只在停车后运行，且只对裁剪/透视矫正后的局部文字 ROI 运行。关闭整页文档方向分类、文档矫正、版面分析等大功能；
 标签确实有旋转时再打开文字行方向分类。
   - 新增/改造中文清洗：保留 [A-Za-z0-9_\-一-鿿]，不再删中文（仅对拉丁做大写规整）。
   - requirements.txt 增加 paddleocr / paddlepaddle（可选块+安装说明；用户在树莓派上 pip 安装）。注意当前小车的
 /home/pi/yolov5-lite 虚拟环境未安装 paddleocr/paddlepaddle，安装后先用实拍 ROI 小图评估耗时和内存。
 2. 粗检测触发（架构约束核心），tag_detector.py 新增 detect_object_presence()：轻量 OpenCV 轮廓/方框判定"ROI
 内是否有物体"；巡逻循环每步抓一帧调用它。
   - 默认 OBJECT_DETECTOR=opencv，作为真实主链路。
   - 可选 OBJECT_DETECTOR=yolov5_lite_cpu：复用 /home/pi/YOLOv5-Lite-master/weights/v5lite-e.pt，只做低频复核，不做每帧
 行驶触发。实测 v5lite-e 在 Pi 5 CPU 上 320 输入单图约 0.54s，太慢则自动降级回 OpenCV。
   - 可选 OBJECT_DETECTOR=hailo_yolo：仅当 hailortcli、/dev/hailo*、rpicam-apps Hailo demo 和 HEF 模型全部验证通过后启用。
 Hailo 优先跑官方 rpicam YOLOv8/YOLOv5 冒烟测试；后续自训练模型选 HAILO8L 支持的 nano 级模型，类别限制为
 card/shelf_slot/box_or_object/unknown_object。
 3. 减速→停车→识别，runtime.py：
   - 新增配置 object_trigger_enabled(默认 True)、object_detector(opencv/yolov5_lite_cpu/hailo_yolo)、object_roi、粗检测阈值、
 slow_speed、object_presence_confirm_frames、object_presence_cooldown_seconds、object_yolo_min_interval_seconds。
   - 主循环：粗检测命中→（可选降一档速逼近）→ motion.stop() → 停稳后 _scan_visible_shelf() 跑完整识别。
   - scan_enabled 默认改 True；run_continuous_patrol 启动文案更新为"检测到目标即停车识别"。
 4. 停稳后完整识别 + 无 tag 也出结果：
   - _detect_frame 解耦：AprilTag 循环之外，无 tag 时对 ROI 跑 _classify_image_region + _try_ocr_text +
 颜色，产出合成检测（tag_id=None、合成分组键如 "OBJ"，带 image_class/ocr_text/color），保证 tag/无 tag

 问题五（新增功能）：普通行驶时用加速度/陀螺仪实时测角、偏移即纠正

 目标

 直线短步前进时，用 MPU6050 陀螺仪积分实时跟踪航向偏差；偏出阈值即用短脉冲反向纠正，保持直线、防跑偏。

 算法（比例脉冲 + 死区，务实匹配厂商粗粒度电机 API）

 - 只有 6 个运动原语（前进/后退/左右平移/左右自转/停），无差速混合，故用"前进小块 + 按需自转纠偏脉冲"交织实现直线闭环。
 - 参考航向在每次转向后与巡逻启动时清零（_turn_90 成功后、run_continuous_patrol
 开始），直线段内不清零，对整段保持真实航向。

 改动

 1. mpu6050.py 新增持久化直线航向跟踪器 StraightHeadingGuard（或等价函数式 API）：
   - 工厂 open_straight_heading_guard()：打开一次陀螺仪、initialize()、_cached_gyro_bias() 取零偏；硬件失败返回
 None（优雅降级）。
   - reset()：航向偏差清零、时间戳复位。
   - update() -> float：读校正后偏航轴速率（复用 _read_corrected_gyro_dps +
 _yaw_axis/_yaw_sign/_yaw_deadband_dps），rate*dt 积分，返回累计航向偏差（度）。
   - 复用现有零偏缓存与偏航轴配置，不每次重开总线。
 2. runtime.py 在 _run_timed_motion 分块前进循环里做航向保持（仅直线巡逻步启用，watch_boundary=True 分支；避障
 watch_boundary=False 不启用）：
   - 每块前 dev = guard.update()；abs(dev) > heading_hold_tolerance_deg（默认 3°）时，插入一段反向自转纠偏脉冲，时长
 clamp(gain*abs(dev), min, max)（默认 gain≈0.02s/°、0.03–0.12s），方向由 dev 符号决定（含 heading_hold_invert
 配置，实车校准）。
   - 惰性打开 self._heading_guard（首帧巡逻），None 则静默禁用。
   - 新增配置：heading_hold_enabled(默认 True)、heading_hold_tolerance_deg、heading_hold_gain、heading_hold_min/max_pul
 se_seconds、heading_hold_correction_speed(复用 turn_speed)、heading_hold_invert。
 3. 纠偏脉冲同样经 motion._call_motion（问题一改造后已急停感知）→ 急停时纠偏也瞬停。
 4. 注意：MPU6050 与 /api/status 的 read_motion_sample 共享 I2C
 总线（bus1/0x68），当前已存在跨线程访问；本改动沿用现状，如现场出现读冲突再加锁（列为风险项）。

 关键文件：robot/mpu6050.py、runtime.py

 验证方案

 单元测试（Windows 开发机可跑，用现有 fake 适配器）

 - pytest tests/ -q，重点 test_runtime.py、test_web_runtime_safety.py、test_mpu6050.py。
 - 现有测试用 action_settle_seconds=0、boundary_confirm_samples=1、部分 scan_enabled=True；改默认值后需核对断言（尤其
 test_runtime.py:70 断言 action_settle_seconds==0.45 要同步改为 0.7；scan_enabled 默认翻真需检查连续巡逻相关用例）。
 - 新增测试：
   - 急停：置 motion.request_stop() 后 _call_motion 不驱动、睡眠即返回。
   - 边界时间窗：模拟四路先后（时间差 <window）点亮 → _feed_boundary_window 命中；间隔 >window 不命中。
   - 航向保持：用 fake gyro 喂偏航速率，验证超阈值时产生正确方向/时长的纠偏脉冲。
   - 无 tag 识别：_detect_frame 在无 AprilTag 时也产出带 ocr_text/image_class 的合成检测。

 实车验证（树莓派，RUN_MODE=robot）

 1. 急停：巡逻/转向/避障途中点"立即停止"，电机应 <0.1s 全停且不再抽动；转向中途按下也能立即停。
 2. 边界：以偏快速度直冲和斜冲黑带，四路窗口应稳定捕获并触发列端动作；正常地面全程不误触发。
 3. 巡逻手感：速度 20/步进 0.18/停 0.7，观察是否平顺、不跑偏、利于识别。
 4. 粗检测→停车→识别：行驶中把卡片/方块/货架放入视野，小车应先由 OpenCV 粗检测停车，再在网页看到
 AprilTag/中文 OCR/图像类别/颜色证据并播报（有 tag 与无 tag 各测一次）。随后才测试 YOLOv5-Lite CPU 复核或 Hailo
 YOLO 后端，确认它们不会拖慢急停/边界检测。
 5. 航向保持：直线段人为轻推使其偏航，观察是否自动回正；必要时调 heading_hold_invert/tolerance/gain。

 风险与待办

 - 视觉（PaddleOCR/YOLO/相机/AprilTag）无法在 Windows 端到端验证，须实车确认并安装依赖。
 - 当前实车未检测到 Hailo runtime/device；Hailo YOLO 只能作为后续增强路线，首版必须靠 OpenCV 粗检测闭环。
 - 当前小车剩余磁盘约 7.3GB；安装 PaddleOCR/PaddlePaddle 前先清理无关缓存或确认空间，避免把系统盘打满。
 - 现有 YOLOv5-Lite 权重不是货架/卡片专用模型，且 CPU 单图推理偏慢；不要让它成为行驶中减速/停车的唯一触发。
 - 改默认值（action_settle_seconds、scan_enabled）可能影响现有单测断言，实施时同步修正。
 - 航向保持符号方向、纠偏增益、粗检测阈值需实车标定（均做成可配置）。
 - MPU6050 I2C 跨线程共享，若现场读冲突再加锁。
