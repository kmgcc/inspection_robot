# A/B 跨列中心线巡线实施计划

**状态：** 仅实施计划，尚未改代码。  
**依据：** `docs/REAL_REQUIREMENTS.md`、`docs/四路红外巡线算法指导书.md`、官方 RASPBOT 四路巡线源码、当前 `runtime.py` / `motion.py` / `sensors.py` 实现。  
**目标：** 仅在 `A4 -> B端` 与 `B1 -> A端` 两段跨列转场中启用单条中心线巡线；普通 A/B 列巡逻继续使用现有边界、航向纠偏和巡航方式。  
**巡检顺序：** `A1 -> A2 -> A3 -> A4 -> B3 -> B2 -> B1`。确认 `B4` 不存在，本计划不引入 `B4`。

## 一、只读调查确认

1. 四路红外接口已在 `src/inspection_robot/robot/sensors.py` 中封装：
   - 硬件寄存器为 `0x0a`。
   - 官方源码位序为 `x2, x1, x3, x4`，项目已归一为 `(左外, 左中, 右中, 右外)`。
   - 现有项目语义保持官方约定：`0 = 黑线`，`1 = 白地`。
   - 新巡线算法内部可以转换为“黑 = 1”，但不能改传感器层和普通边界层语义。
2. 高速边界防漏检已在 `src/inspection_robot/runtime.py` 中实现：
   - `_run_timed_motion()` 在运动期间按 `motion_guard_poll_seconds` 高频轮询传感器。
   - `_poll_boundary_during_motion()` 命中后立即 `motion.stop()`，并锁存 `_pending_boundary_state`。
   - `_feed_boundary_window()` 记录每一路最近压黑时间，用时间窗累计“窗口内四路覆盖”，解决斜压和高速掠过时四路不在同一帧全黑的问题。
3. 普通列边界默认是 `boundary_min_black_sensors=1`，已有测试保护“任意一路黑即边界”。这不能为了转场巡线全局改成四路全黑。
4. 现有 `robot/line_following.py` 是离散规则：`forward / strafe / rotate / wait`。它适合旧调试，不适合本次单中心线稳定转场。
5. `motion.move_forward_corrected_slow()` 已支持前进差速：
   - `direction="right"` 时左轮更快、右轮更慢。
   - `direction="left"` 时左轮更慢、右轮更快。
   - 这是转场 PD 巡线第一版应优先使用的底层接口。
6. 航向保持也会调用差速前进。转场巡线期间必须暂停原列航向纠偏，避免 heading hold 与巡线 PD 同时争夺左右轮差速。
7. `A4` / `B1` 已作为跨列相位锚点使用；识别到它们时会暂停移动视觉扫描，直到下一次边界转向完成。

## 二、业务流程

### A 到 B

1. A 列巡逻到 `A4`。
2. A4 末端横向边界触发，普通列模式按现有“任意一路黑即可锁存”逻辑停车。
3. 顺时针转约 90 度。
4. 进入 `A_TO_B` 转场上下文，启用单条中心线巡线。
5. 离开起点横线后解锁终点边界。
6. 沿中心线巡线到 B 端。
7. B 端横向边界触发，立即停车并锁存。
8. 顺时针转约 90 度。
9. 关闭转场巡线，恢复 B 列原有巡逻方式。

### B 到 A

1. B 列巡逻到 `B1`。
2. B1 末端横向边界触发，普通列模式停车。
3. 顺时针转约 90 度。
4. 进入 `B_TO_A` 转场上下文，启用单条中心线巡线。
5. 离开起点横线后解锁终点边界。
6. 沿中心线巡线到 A 端。
7. A 端横向边界触发，立即停车并锁存。
8. 顺时针转约 90 度。
9. 关闭转场巡线，恢复 A 列原有巡逻方式。

## 三、边界模式设计

边界检测必须模式化，但不能复制一套变慢的新机制。

### 普通列模式

保持现状：

1. 任意一路压黑即可进入边界候选。
2. 运动期间仍由 `_run_timed_motion()` 高频轮询。
3. 命中后立即停车并锁存，后续由 `_handle_tape_boundary()` 判定转向或绕行。

### 转场模式

仅在跨列中心线巡线期间启用：

1. 中心线状态如 `0110`、`0100`、`0010`、`1100`、`0011` 属于正常巡线，不是边界。
2. `0000` 是丢线恢复问题，不是边界。
3. 终点边界不是只认同一帧四路全黑。终点已解锁后必须复用 `_feed_boundary_window()` 的窗口语义：
   - 同一帧原始 `(0,0,0,0)` 立即触发。
   - 或者在既有短时间窗内，四颗传感器分别都被观察到压黑，即使没有任何一帧四路同时黑，也触发。
   - 触发后立即锁存、停车，只允许触发一次。
4. 转场终点边界必须使用原始高频传感器状态，不等待巡线去抖，不等待 PD 控制周期。

## 四、起点离场保护

不能使用固定“前 0.2-0.4 秒忽略边界”的死时间窗。正确策略是“离开起点横线后才解锁终点”。

流程：

1. 完成第一次顺时针 90 度转向。
2. 建立转场上下文，进入 `LEAVING_START_BOUNDARY`。
3. 此阶段不累计终点边界候选，也不使用起点横线留下的四路黑历史。
4. 连续观察到若干帧有效、非 `1111` 的中心线状态后，认为已离开旧横线并进入跨列中心线。
5. 清空转场专用边界窗口和候选状态。
6. 解锁终点边界。
7. 从这一刻开始重新累计终点四路黑事件。

第一版建议确认条件：

1. 归一化“黑 = 1”后，连续 `transfer_line_exit_confirm_frames` 帧属于中心线有效集合。
2. 有效集合初始为：`0110`、`0100`、`0010`、`1100`、`0011`、`1110`、`0111`。
3. 不把 `1111` 作为离场确认。
4. 确认帧数先配置为 2-3 帧，后续上车调参。

## 五、转场上下文

不要只依赖 `_last_shelf_anchor` 推断当前转场。第一次转向前应锁存一个简洁上下文。

建议字段：

```text
origin_anchor: A4 / B1
direction: A_TO_B / B_TO_A
destination_column: B / A
transfer_heading: 第一次转向后的跨列行驶朝向
post_turn_heading: 第二次转向后恢复列巡逻的朝向
endpoint_unlocked: bool
endpoint_latched: bool
```

用途：

1. 边界回调知道当前处于哪一次转场。
2. 视觉暂停、终点转向、恢复巡逻不受 `_last_shelf_anchor` 更新顺序影响。
3. 日志可以清楚记录 `A_TO_B` 或 `B_TO_A`。

## 六、巡线控制设计

第一版不再扩大旧离散 `decide_line_follow_motion()` 的职责，而是在 `robot/line_following.py` 中新增连续巡线能力，并保留旧接口给测试模式使用。

### 数据分层

1. 原始状态：项目现有语义 `(0=黑, 1=白)`，用于高频终点边界锁存。
2. 巡线状态：内部转换为 `(1=黑, 0=白)`，用于去抖、位置估计和 PD 控制。

### 位置估计

传感器坐标：

```text
LO = -1.5
LI = -0.5
RI = +0.5
RO = +1.5
```

命中黑线时用加权平均得到 `error_raw`。

### 控制输出

1. 使用 PD，不启用积分项。
2. `error > 0` 表示黑线在右侧，应向右修正。
3. 输出到 `move_forward_corrected_slow(speed=base_speed, correction=turn, direction=left/right)`。
4. 偏差越大，基础速度越低。
5. 只剩外侧单颗命中时进一步降速。

### 第一版状态

第一版必须实现：

1. `TRACK`：正常循线。
2. `BRIDGE`：短暂 `0000` 时低速沿最后趋势桥接。
3. `BIASED_SEARCH`：桥接超时后朝最后看到线的一侧低速找线。
4. `FAILSAFE`：长时间找不到线后停车。

后置：

1. `SWEEP_SEARCH`：左右扫描找线。
2. `REACQUIRE`：重新找到线后的平滑接管。

这样避免第一版为了形式完整而过度设计。

## 七、Runtime 接入优先级

运行时优先级：

1. 急停、手动接管。
2. 障碍安全处理。
3. 转场终点边界原始高频窗口锁存。
4. 转场巡线状态机。
5. 普通列巡逻逻辑。

高频边界监测必须覆盖整个转场运动过程。PD 控制只决定下一小段运动参数，不是边界事件的前置条件。只要电机在动，就应继续通过 `_run_timed_motion()` 的 guard 轮询原始传感器，并可直接 `stop()`。

转场巡线期间：

1. `heading_hold=False`。
2. `watch_boundary=True`，但边界 predicate 使用转场模式。
3. `keep_running` 是否开启需结合 tick 时长测试决定；安全优先，第一版可保持短 tick 但不降低 guard 频率。
4. 转场终点锁存后停止巡线 PD，执行第二次顺时针转向。
5. 第二次转向完成后调用现有 heading guard reset / ZUPT 逻辑，恢复普通列巡逻。

## 十、测试计划

### 单元测试

1. `0110` 误差接近 0，输出直行或极小修正。
2. `0100 / 1100` 输出向左修正。
3. `0010 / 0011` 输出向右修正。
4. `0000` 从 `TRACK` 进入 `BRIDGE`，不立即作为边界。
5. `BRIDGE` 超时后进入 `BIASED_SEARCH`。
6. 超过 failsafe 时输出停车。

### Runtime 集成测试

必须覆盖：

1. A4 普通列边界触发后，第一次顺时针转向并进入 `A_TO_B` 转场上下文。
2. B1 普通列边界触发后，第一次顺时针转向并进入 `B_TO_A` 转场上下文。
3. 起点横线残留 `1111` 期间，终点未解锁，不触发第二次转向。
4. 连续确认有效非 `1111` 中心线后，清空转场专用边界窗口并解锁终点。
5. 中心线稳定 `0110` 绝不触发终点。
6. 解锁后同一帧原始 `(0,0,0,0)` 立即触发终点一次。
7. 解锁后 LO、LI、RI、RO 在短窗口内依次压黑，但没有任何一帧四路同时黑，仍触发终点一次。
8. 起点横线期间出现的全黑历史，不得在后续解锁后补触发。
9. 终点触发后只允许一次锁存，不能重复转向。
10. 到达对侧横线并完成第二次顺时针转向后，关闭巡线并恢复普通列巡逻。
11. 普通 A/B 列“任意一路黑即边界”逻辑不回归。
12. 转场期间视觉扫描仍暂停，终点转向后恢复。
13. 转场期间障碍、急停、手动接管优先级仍高于巡线。

### 上车测试

1. 架空或手推：
   - 验证原始 mask、内部黑白极性、error、state、endpoint unlock、endpoint latch。
2. 低速单段 A4 -> B：
   - 只验证第一次转向、离场保护、中心线巡线、终点锁存、第二次转向。
3. 低速单段 B1 -> A：
   - 验证返程对称性。
4. 完整闭环：
   - A 列巡逻 -> A4 转场 -> B 列巡逻 -> B1 转场 -> A 列巡逻。
5. 提速验证：
   - 专门验证高速斜压横线时，窗口内四路覆盖仍不漏检。

## 十一、不可破坏行为

1. 不增加 `B4`。
2. 不改变 `sensors.py` 的黑白语义。
3. 不降低普通列边界灵敏度。
4. 不把转场终点退化成“必须同一帧四路全黑”。
5. 不让巡线去抖阻塞终点边界快速锁存。
6. 不让 heading hold 与转场 PD 同时控制差速。
7. 不让视觉扫描、OCR、语音等任务阻塞转场边界锁存。
8. 不让丢线搜索无限运行；真正找不到线必须安全停车。

## 十二、阶段划分

### 阶段 1：边界模式与上下文

目标：

1. 引入转场上下文。
2. 抽象普通列 / 转场终点两种边界触发条件。
3. 转场终点复用 `_feed_boundary_window()` 的窗口内四路覆盖语义。

风险：

1. 普通列任意一路边界退化。
2. 起点横线历史误触发终点。

验证：

1. 普通列边界回归测试。
2. 起点历史清空测试。
3. 错峰四路黑触发测试。

### 阶段 2：连续巡线控制器

目标：

1. 实现位置估计、PD、速度调度。
2. 实现 `TRACK / BRIDGE / BIASED_SEARCH / FAILSAFE`。

风险：

1. 差速方向反。
2. 参数过激导致蛇形。

验证：

1. 算法单测。
2. 架空/手推日志。

### 阶段 3：Runtime 转场接入

目标：

1. A4/B1 第一次转向后进入转场巡线。
2. 转场期间暂停 heading hold。
3. 终点锁存后第二次转向并恢复普通列。

风险：

1. 与 cruise scanner 状态冲突。
2. 手动/急停/障碍优先级被巡线占用。

验证：

1. Runtime 集成测试。
2. 视觉暂停测试。
3. 手动/急停/障碍优先级测试。

### 阶段 4：上车调参

目标：

1. 低速跑通 A_TO_B 和 B_TO_A。
2. 调整 `Kp/Kd/speed/turn_max`。
3. 验证高速横线不漏检。

风险：

1. 传感器高度、地面反光、胶带宽度造成 mask 抖动。
2. 电机低速死区导致差速无效。

验证：

1. 转场日志。
2. 完整闭环视频。
3. 高速斜压横线专项测试。
