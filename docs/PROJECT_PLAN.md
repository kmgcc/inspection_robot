# 基于 AprilTag 的巡线式物品巡检与人工回收闭环系统全局规划书

## 结论先行

本项目最终建议做成一个可在 6 天内完成、答辩时逻辑清楚且有展示效果的工程系统：

**基于 AprilTag 货位标签的巡线式物品巡检、异常上报与人工回收确认系统。**

它的基本画面是：小车沿黑线或固定路线巡检若干分区，车载摄像头识别贴在物品方块或货位上的 AprilTag 标签，系统把识别到的标签 ID 映射为物品类别和分区状态；如果发现物品缺失、错放、重复、未回收等异常，小车用蜂鸣器和 RGB 灯告警，同时在本地网页看板上记录事件；人工取走异常物品或完成回收后，通过车上 KEY1、网页按钮或确认二维码完成闭环确认。

这个题目比“自由避障加自动回收”更稳。官方资料已经证明，RASPBOT V2 有 AprilTag 识别、巡线、超声波停车恢复、蜂鸣器、RGB 灯、OLED、云台、ROS2 底盘话题等基础能力；但资料没有证明它带机械臂或夹爪，也没有证明它具备自由空间动态绕障能力。所以我们的系统边界应当写清楚：**小车负责巡检、发现、告警、上报和确认；人工负责实际回收动作。** 这不是降级，而是把项目从“演示概念”拉回“能交付的闭环系统”。

### 4. 巡线与避障边界

官方资料支持巡线、停车恢复和规则式避障，但不支持真正意义上的自由空间绕障。

- `05.小车综合玩法/1.小车红外巡线/1.小车红外巡线.pdf` 第 1-3 页：通过四路巡线模块实现小车巡黑线。
- `19.程序源码汇总/程序源码汇总.zip` 内 `project_demo/05.Comprehensive_gameplay/1.infrared_patrol_line.ipynb`：第 29-36 行读取四路巡线传感器，第 47-82 行按左右传感器状态前进、左转、右转。
- `05.小车综合玩法/4.红外巡线+超声波避障/4.红外巡线+超声波避障.pdf` 第 1-3 页：巡线过程中检测到障碍物，小车停止并蜂鸣，障碍物拿开后继续巡线。
- 同 zip 内 `project_demo/05.Comprehensive_gameplay/4.linetrack_ultraavoid.ipynb`：第 48-55 行读取超声波距离，距离小于 200 mm 时 `stop_robot()` 并打开蜂鸣器。
- `05.小车综合玩法/2.超声波避障/2.超声波避障.pdf` 第 1-2 页：当障碍物小于设定距离时，小车左旋避开障碍物。
- 同 zip 内 `project_demo/05.Comprehensive_gameplay/2.ultrasonic_obstacle_avoidance.ipynb`：第 28-29 行定义距离阈值，第 43-55 行实现近距离后退、中距离停车左旋、远距离前进。

因此，本项目中的“动态避障”应表述为：**动态障碍检测、停车告警、障碍解除后恢复巡检**。若要展示“绕行”，只做固定支路或固定动作，不承诺自由路径规划。

## 二、项目目标

### 1. 必须完成的目标

1. 小车能沿黑线或预设路线低速巡检。
2. 小车能识别 AprilTag 标签 ID，并把 ID 映射为物品名、所属分区和预期状态。
3. 系统能记录每次巡检结果，包括时间、分区、标签 ID、物品名、状态、截图或帧编号。
4. 系统能发现至少三类异常：缺失、错放、未确认回收。
5. 异常发生时，小车能蜂鸣器告警，RGB 灯显示异常颜色。
6. 本地网页看板能实时显示巡检状态、异常列表、人工确认按钮和历史日志。
7. 人工完成回收后，能通过网页按钮、KEY1 或确认二维码完成确认，系统把异常改为已处理。

### 2. 加分目标

1. 网页看板显示小车当前阶段：巡检中、识别中、异常等待确认、障碍等待解除、任务完成。
2. 支持任务优先级，例如高优先级物品异常时持续告警，普通异常只记录。
3. 支持分区地图简图，用 A/B/C/D 表示不同巡检区域。
4. 支持导出 CSV 日志，便于答辩展示数据沉淀。
5. 支持障碍解除后自动恢复巡线。

### 3. 明确不做的内容

1. 不做 SLAM。
2. 不训练深度学习模型。
3. 不做自由空间路径规划。
4. 不承诺动态自由绕障。
5. 不做机械臂抓取或夹爪回收，因为官方资料没有证明本车具备可执行抓取的硬件。
6. 不把大模型作为核心链路。它可以在答辩里作为扩展方向，不进入 6 天必做范围。

## 三、系统总体架构

建议采用“一个巡检主程序 + 一个网页看板 + 一个数据文件”的小系统结构。它不复杂，但链路完整。

```text
摄像头 /dev/video0
    |
    v
AprilTag 识别模块
    |
    v
巡检状态机 ---- 四路巡线模块
    |          |
    |          v
    |       底盘控制
    |
    +---- 超声波障碍检测
    |
    +---- 蜂鸣器 / RGB 告警
    |
    v
巡检事件记录 store.json 或 sqlite
    |
    v
Flask 本地网页看板
```

### 1. 主路线

主路线优先做成 Python 单进程或双线程程序：

- 摄像头：`cv2.VideoCapture(0)`，与官方视觉例程一致。
- 标签识别：优先使用 `dt_apriltags`，识别 `tag36h11`。
- 底盘与模块：复用 `Raspbot_Lib` 或 `McLumk_Wheel_Sports`，与官方 Jupyter 示例一致。
- 网页看板：使用 Flask，页面每 1 秒轮询 `/api/status`。
- 数据存储：先用 JSON 文件或内存列表，答辩前再导出 CSV。

这条路线的好处是调试直接，网页和巡检逻辑都在 Python 内，适合 6 天课程项目。

### 2. 兜底路线

如果车上 `dt_apriltags` 或 Python 驱动库环境不顺，就走 ROS2 兜底：

- 进入 Docker：参考 `17.Docker/5、进入机器人的docker容器.pdf`。
- 启动底盘：`ros2 launch yahboomcar_bringup bringup.launch.py`。
- 启动 AprilTag 识别：`ros2 run yahboomcar_apriltag apriltag_identify`。
- 使用 ROS2 话题 `single_apriltag_id` 获取 ID，或先用官方窗口演示识别，系统主程序只接收人工/终端输入 ID 做看板闭环。

兜底路线牺牲自动化程度，但能保证答辩时 AprilTag 官方能力不丢。

## 四、功能模块规划

### 1. 标签字典模块

建立一个本地配置，例如：

```python
TAG_MAP = {
    0: {"name": "Apple", "zone": "A", "expected": "present", "priority": 2},
    1: {"name": "Milk", "zone": "A", "expected": "present", "priority": 1},
    2: {"name": "Book", "zone": "B", "expected": "present", "priority": 3},
    3: {"name": "RecycleBox", "zone": "C", "expected": "confirm_only", "priority": 2},
}
```

这里要注意，AprilTag 官方只保证识别 ID 和位置，不保证识别物品类别。物品类别由我们自己通过 ID 映射出来。答辩时必须这样讲，反而显得技术边界清楚。

### 2. 巡检状态机

建议先做有限状态机，不要一上来写复杂调度：

```text
INIT
  -> PATROL
  -> TAG_DETECTED
  -> NORMAL_LOGGED
  -> PATROL

PATROL
  -> OBSTACLE_WAIT
  -> PATROL

TAG_DETECTED
  -> ABNORMAL_ALARM
  -> WAIT_RECOVERY_CONFIRM
  -> RECOVERY_CONFIRMED
  -> PATROL

PATROL
  -> FINISHED
```

状态含义：

- `INIT`：初始化摄像头、驱动库、网页看板数据。
- `PATROL`：沿黑线巡航，同时低频检测 AprilTag 和超声波。
- `TAG_DETECTED`：识别到标签，暂停或减速，判断所在分区和物品状态。
- `NORMAL_LOGGED`：正常物品只记录日志，不告警。
- `ABNORMAL_ALARM`：异常物品停车、蜂鸣、RGB 变红。
- `WAIT_RECOVERY_CONFIRM`：等待人工处理。
- `RECOVERY_CONFIRMED`：人工确认后关闭告警，记录闭环。
- `OBSTACLE_WAIT`：超声波检测到障碍物，停车等待。
- `FINISHED`：完成一轮巡检。

### 3. 分区巡检模块

6 天内不要做真实二维坐标。建议用下列两种方式之一：

1. **站点标签法**：每个分区入口放一个 AprilTag 作为分区标签，如 `Zone-A`、`Zone-B`。识别到分区标签后，之后识别到的物品都归入该分区。
2. **路线计数法**：每经过一个停止点或路口，计数加一，映射到 A/B/C/D 分区。

优先推荐站点标签法。它更稳，也更容易在答辩时解释：小车不是盲目猜位置，而是通过视觉标签完成分区定位。

### 4. 异常判定模块

异常规则先做简单：

| 异常类型 | 判定方式 | 展示话术 |
|---|---|---|
| 缺失 | 到达某分区后，规定时间内没识别到应有标签 | A 区 Apple 未发现 |
| 错放 | 在 A 区识别到配置属于 B 区的标签 | Book 出现在错误分区 A |
| 未回收确认 | 异常已告警，但人工未确认 | 异常等待处理 |
| 重复出现 | 同一轮巡检中同一标签被不同分区记录 | 标签重复，需要人工核查 |

这四类足够支撑答辩，不必再扩展复杂库存模型。

### 5. 声光告警模块

告警规则建议如下：

- 正常识别：RGB 绿色闪一下。
- 普通异常：RGB 黄色，蜂鸣一次。
- 高优先级异常：RGB 红色，蜂鸣三次，并保持看板红色状态。
- 人工确认后：RGB 绿色，蜂鸣一次，回到巡检。
- 障碍物：RGB 蓝色或紫色，蜂鸣短促提示，障碍解除后关闭。

官方依据：

- 蜂鸣器：`03.小车基础教程/01.蜂鸣器控制.pdf` 第 1 页。
- RGB：`03.小车基础教程/03.RGB控制.pdf` 第 1-4 页。

### 6. 网页看板模块

网页看板是本项目的展示重点。建议页面只做一个主界面，避免把时间耗在复杂前端。

页面区域：

```text
顶部：系统标题 + 当前状态 + 当前轮次
左侧：分区卡片 A/B/C/D，显示正常/异常/待确认
中间：实时事件流
右侧：异常列表和确认按钮
底部：最近识别标签、障碍物距离、告警状态、导出日志按钮
```

推荐接口：

```text
GET  /                 看板页面
GET  /api/status       当前状态
GET  /api/events       事件列表
POST /api/confirm      人工确认某条异常
POST /api/start        开始巡检
POST /api/stop         停止巡检
GET  /api/export.csv   导出日志
```

推荐数据结构：

```json
{
  "run_id": "2026-07-01-001",
  "state": "WAIT_RECOVERY_CONFIRM",
  "current_zone": "A",
  "last_tag": {
    "id": 0,
    "name": "Apple",
    "zone": "A",
    "confidence": "detected"
  },
  "obstacle": {
    "distance_mm": 180,
    "blocked": true
  },
  "alarms": [
    {
      "id": "evt-0007",
      "type": "wrong_zone",
      "message": "Book appeared in Zone A, expected Zone B",
      "priority": 3,
      "status": "waiting_confirm"
    }
  ]
}
```

看板实现建议：

- 后端用 Flask。
- 前端用原生 HTML/CSS/JS 即可。
- 每 1000 ms 拉取一次 `/api/status`。
- 页面颜色保持克制：正常绿色，异常红色，等待黄色，障碍蓝色。
- 不要做炫技动画。答辩看的是系统闭环，不是网页花哨程度。

## 五、6 天推进步骤

### 第 1 天：环境摸底与最小闭环

目标：确认车能动、摄像头能读、网页能开。

任务：

1. 启动车，连接 Wi-Fi 或局域网。
2. 按 `03.小车基础教程/00.开发前的准备/开发前的准备.pdf` 关闭开机自启动大程序，避免占用设备。
3. 运行官方电机、蜂鸣器、RGB、超声波、巡线示例，确认硬件状态。
4. 运行官方 AprilTag 识别例程，确认标签 ID 可识别。
5. 在电脑或树莓派上跑一个最小 Flask 页面，显示“系统在线”。

交付物：

- 硬件状态表。
- 能识别至少 1 个 AprilTag 的截图或录屏。
- 能访问的网页看板空壳。

退出标准：

- 摄像头 `/dev/video0` 可用。
- 至少 1 个标签能稳定读出 ID。
- 蜂鸣器和 RGB 能被程序控制。

### 第 2 天：标签识别与配置表

目标：把 AprilTag ID 转换成物品和分区信息。

任务：

1. 整理标签 ID，给每个标签贴上物品名和期望分区。
2. 写 `tag_config.py` 或 `tag_config.json`。
3. 写一个独立识别脚本，只做摄像头读取、AprilTag 检测、ID 显示、去重。
4. 将识别结果推送到看板 `/api/status`。

交付物：

- 标签配置表。
- 标签识别脚本。
- 看板能显示最近识别标签。

退出标准：

- 至少 4 个标签能被识别。
- 同一个标签不会在 1 秒内重复刷屏。
- 看板能显示 ID、物品名、分区。

### 第 3 天：巡线与障碍停车恢复

目标：让小车能沿路线巡检，并在障碍出现时停车。

任务：

1. 复用官方红外巡线逻辑，先低速跑直线和简单弯道。
2. 接入超声波距离检测，小于阈值时停车并告警。
3. 障碍物移开后恢复巡线。
4. 将状态同步到看板：`PATROL`、`OBSTACLE_WAIT`、`PATROL_RESUMED`。

交付物：

- 巡线脚本。
- 障碍停车恢复演示视频。
- 看板障碍状态显示。

退出标准：

- 小车能完成至少 1 条简单路线。
- 人手或纸盒挡在前方时，小车停车。
- 移开障碍后，小车能继续巡线。

### 第 4 天：异常判定与声光告警

目标：实现项目的核心工程逻辑。

任务：

1. 写异常判定函数：缺失、错放、未确认、重复。

## 七、关键实现建议

### 1. AprilTag 去重

标签识别会连续多帧读到同一个 ID，所以必须去重：

```python
last_seen = {}

def should_accept(tag_id, now, cooldown=1.5):
    if tag_id not in last_seen:
        last_seen[tag_id] = now
        return True
    if now - last_seen[tag_id] > cooldown:
        last_seen[tag_id] = now
        return True
    return False
```

### 2. 异常事件不可覆盖

每个异常都应生成独立事件 ID，不要只在状态里保存最后一个异常。否则答辩时看板会丢历史。

```python
event = {
    "id": f"evt-{counter:04d}",
    "time": now_iso(),
    "type": "wrong_zone",
    "tag_id": tag_id,
    "item": item_name,
    "zone": current_zone,
    "expected_zone": expected_zone,
    "status": "waiting_confirm"
}
```

### 3. 先停车再识别

移动中识别 AprilTag 容易抖。建议识别到疑似标签后先减速或停车 0.5 秒，再连续读 3 帧，取出现次数最多的 ID。

### 4. 固定光照和固定距离

AprilTag 虽然比普通图像识别稳，但它依赖清晰边缘。演示时应让标签正对摄像头，距离先固定在 20-60 cm。官方追踪/跟随资料也提醒，标签要对着摄像头，移动速度不能太快。

### 5. 网页确认优先于按键确认

KEY1 很适合做硬件确认，但网页按钮更适合答辩展示。建议先实现网页确认，再把 KEY1 作为加分项。

## 八、答辩演示流程

### 标准演示

1. 打开网页看板，点击“开始巡检”。
2. 小车沿黑线前进，看板状态为“巡检中”。
3. 小车识别 A 区正常标签，例如 Apple，RGB 绿灯闪烁，看板记录正常事件。
4. 小车识别一个错放标签，例如 B 区的 Book 出现在 A 区。
5. 小车停车，蜂鸣器告警，RGB 红灯，看板出现异常事件。
6. 人工取走错放物品，点击网页“确认已回收”。
7. 看板状态变为“已确认”，小车继续巡检。
8. 放置障碍物，小车进入“障碍等待解除”；移开后恢复巡线。
9. 巡检完成，导出日志。

### 答辩讲解重点

1. 我们没有把系统做成单一视觉 demo，而是做成巡检闭环。
2. AprilTag 提供稳定 ID，物品类别通过配置映射，避免训练模型。
3. 分区采用站点标签或路线状态，不做虚假的高精度定位。
4. 遇障采用停车恢复，符合官方例程边界。
5. 回收采用人工确认，符合实际硬件约束。
6. 网页看板把机器人状态、异常和人工处理过程可视化，这是工程系统的完整性。

## 九、风险与应对

| 风险 | 可能表现 | 应对 |
|---|---|---|
| AprilTag 识别不稳 | 远距离、反光、倾斜时漏识别 | 固定距离，停车识别，多帧投票，增大标签 |
| 巡线不稳 | 过弯丢线、地图脏、光照变化 | 低速，粗黑线，固定路线，准备手动遥控兜底 |
| 超声波误报 | 近距离抖动 | 连续 3 次低于阈值才停车 |
| 网页卡顿 | 看板不刷新 | 终端日志兜底，网页只做状态展示 |
| ROS2/Python 环境冲突 | 依赖找不到 | 主路线失败就用官方 ROS2 节点演示 AprilTag，主程序做看板和日志 |
| 现场网络不稳定 | 浏览器打不开 | 手机/电脑提前连小车热点，准备录屏备份 |
| 电量不足 | 巡线速度变慢，转向失败 | 答辩前充满电，路线缩短 |

## 十一、最终建议

项目名称建议定为：

**基于 AprilTag 的移动式物品巡检、异常上报与人工回收确认系统**

这个名字比“动态避障回收机器人”更准确。它承认真实硬件边界，同时保留三个明确亮点：第一，视觉标签让物品身份可验证；第二，网页看板让巡检过程可追踪；第三，人工确认让异常处理形成闭环。6 天时间里，真正值得追求的不是把功能堆满，而是让每个功能都能跑通、能解释、能复现。

答辩时就围绕一句话讲：**我们把官方零散例程组合成一个完整的巡检业务系统，小车负责发现问题，网页负责上报问题，人工确认负责关闭问题。**
