# 固定货架通道巡逻小车全局规划书

**当前权威需求：** 以 `docs/REAL_REQUIREMENTS.md` 为准。本文是对总计划的同步版，用于替代旧的“固定栅格地图 + A* 路径规划”主叙事。

## 一、项目结论

本项目不是循线小车，也不是让小车沿黑线行驶。真实目标是：

**在固定货架通道中，让 RASPBOT V2 麦克纳姆轮小车自动往返巡逻，侧向扫描货架与物品，发现缺货、错放、重复、未知物品或证据冲突后，用灯光、声音和网页看板进行提示、上报和人工确认。**

小车开机或网页点击“开始”后，从人工摆放好的初始位置慢速向前行驶。摄像头通过云台初始化到机身侧面，用于扫描与行驶方向垂直的货架。小车不依赖黑线作为路线；黑胶带只用于列端触发、安全边界和禁区兜底。

当前明确的场地是：A 列货架按行驶方向依次为 `A1`、`A2`、`A3`、`A4`，每处都有货物。小车行驶到货架列尽头时会遇到横向黑胶带，四路底部传感器同时检测到黑色，表示到达列端或禁区触发区。此时小车停车，顺时针原地旋转 90 度，然后继续前进。每列货架尽头使用“工字形”黑胶带，让小车在 A 列和 B 列等通道之间往返循环。

网页上的地图不应是预先画死的栅格地图。真实系统初始没有完整地图，只有运行过程中识别到货架、物品、边界、转向点和异常后，软件逐步生成巡检拓扑、路线图或事件时间线。

## 二、项目边界

必须坚持：

1. 不做 SLAM。
2. 不做开放环境自由自动驾驶。
3. 不训练大型通用识别模型。
4. 不做机械臂抓取或搬运。
5. 不让 LLM 控制底盘、电机、避障或转向。
6. 不把黑胶带当作主巡线路径。
7. 不把旧固定栅格地图或 A* 路径规划写成真车主链路。

A*、Dijkstra 或固定栅格地图可以保留为软件兜底演示、答辩辅助说明或未来扩展，但当前真车主链路是“货架通道循环巡逻 + 端点触发转向 + 侧向识别 + 规则比对”。

## 三、真实运行流程

1. 人工把小车放到初始位置。
2. 系统启动网页服务，必要时开机自启。
3. 官方 Yahboom App 主程序被禁用或启动前清理，避免占用摄像头、底盘、I2C 和端口。
4. 云台初始化到侧向货架视角。
5. 小车慢速向前巡逻。
6. 侧向摄像头识别货架 AprilTag，更新当前货架。
7. 识别到货架时播放货架提示音。
8. 识别到物品时播放物品提示音。
9. 第一轮巡逻只记录观察，不上报缺货。
10. 第二轮及之后开始根据货架清单判断缺失、重复、错放、未知和证据冲突。
11. 检测到缺货等高优先级异常时，红色 LED 提示，并播放语音警报。
12. 小车到达列端，四路传感器全黑时，顺时针原地转 90 度并继续。
13. 遇到前方障碍时停车等待 6 秒；障碍解除则恢复，障碍仍在则执行绕行。
14. 绕行过程可被新障碍打断，并进入嵌套避障。
15. 网页实时显示运行模式、硬件连接、当前轮次、当前货架、最近识别、告警、事件和动态生成的巡检拓扑。

## 四、场地与黑胶带

黑胶带必须区分两种语义：

1. **四路全黑：** 正常列端触发。动作是停车、顺时针旋转 90 度、继续前进。
2. **非预期压线或局部禁区：** 安全兜底。处理逻辑与障碍类似，先停、等待、必要时绕行或报警。

旧文档中的“任意一路压黑就后退 5-10 cm 并按地图规避”不再是主逻辑。新的判断必须优先识别“四路全黑列端触发”。

## 五、识别与异常规则

货架识别：

1. 货架通过 AprilTag 识别。
2. 货架文字如 `A1`、`A2` 可作为 OCR 补充校验。
3. AprilTag 是货架身份主来源。

物品识别需要支持：

1. AprilTag。
2. 文字识别。
3. 图形识别。
4. 可选颜色识别，如蓝、绿、黄、红。

如果没有颜色信息，系统不能报错，应记录为空、未知或未提供。颜色只作为物品价值、属性或复核证据，不应单独推翻 AprilTag 主身份。

推荐算法方向采用“传感器触发 + 视觉确认 + OCR 读取 + 语音反馈”，不把所有判断压给一个大模型。传感器负责发现障碍、列端和扫描触发；AprilTag 负责货架或卡片身份、位置、角度和稳定性；轻量目标检测只判断是否存在卡片/货架槽位/未知物体以及是否进入扫描 ROI；OpenCV 做透视矫正和颜色补充；PaddleOCR 轻量中文模型读取局部文字；Piper 或固定音频负责离线播报。

硬件建议是 Raspberry Pi 5 8GB + AI HAT+ 13 TOPS + Camera Module 3 标准视角版 + 主动散热。第一阶段可以不训练模型，先用 AprilTag + 卡片矩形轮廓 + OCR 完成稳定演示；第二阶段再训练少数类别的轻量检测模型，例如 `card`、`shelf_slot`、`unknown_object`，低频运行在 AI HAT+ 上。

异常类型：

1. `missing_item`：预期有但未识别到。
2. `duplicate_item`：同一货架重复识别。
3. `wrong_shelf`：物品出现在错误货架。
4. `unknown_item`：物品不在配置中。
5. `evidence_mismatch`：AprilTag、文字、图形、颜色之间冲突。

## 六、音频与灯光

音频至少分三类：

1. 扫描到货架时播放货架提示音。
2. 扫描到具体物品时播放物品提示音。
3. 检测到缺货或高优先级异常时播放语音警报。

当前固定音效已经放入 `src/inspection_robot/static/audio/`：

| Cue | 文件 | 触发 |
|---|---|---|
| `obstacle` | `obstacle.wav` | 检测到障碍物或非预期禁区 |
| `first` | `first.wav` | 检测到货架 |
| `following` | `following.wav` | 检测到货架上的物品，每个物品播放一次 |

第一阶段使用这些固定 WAV 音效；后续缺货语音可以继续接固定音频或 Piper TTS。音频播放必须通过树莓派默认音频设备非阻塞输出，失败时记录事件，但不能影响巡逻主流程。

灯光建议：

1. 正常巡逻：绿色或低亮度状态。
2. 货架扫描：绿色短闪。
3. 物品识别：蓝色或白色短闪。
4. 障碍等待：蓝色或紫色。
5. 缺货、错放、高优先级异常：红色。

## 七、避障与禁区

障碍逻辑：

1. 超声波检测到前方障碍。
2. 停车。
3. 等待 6 秒。
4. 6 秒后再次检测。
5. 障碍消失则恢复前进。
6. 障碍仍在则绕行。

右侧绕行动作：

1. 顺时针原地旋转 90 度。
2. 前进一个车身距离。
3. 逆时针旋转 90 度。
4. 前进一个车身距离。
5. 再次逆时针旋转 90 度。
6. 前进一个车身距离。
7. 顺时针旋转 90 度，恢复原方向。

绕行过程必须可打断。如果绕行中遇到新障碍，进入新的等待和绕行流程；需要设置最大嵌套次数或安全超时，避免无限绕行。

非预期禁区按障碍逻辑处理。列端四路全黑不按障碍处理，而按正常端点转向处理。

## 八、网页与运维

网页必须支持：

1. 显示 `simulate` 或 `robot` 运行模式。
2. 显示是否连接真实硬件。
3. 开始巡逻。
4. 急停。
5. 停止或暂停。
6. 前进、后退、左移、右移。
7. 左转 90 度、右转 90 度。
8. 云台初始化。
9. 音频测试或预留播报。
10. 动态巡检拓扑或轨迹显示。
11. 当前轮次和第一轮跳过检测状态。
12. 事件列表和待确认异常。

网页不得再把一个预设栅格称为真实“当前地图”。如果尚未生成拓扑，应显示“尚未生成巡检地图”或等价空状态。

运维必须保证：

1. 彻底禁用官方 App 主程序，但不破坏 Raspberry Pi 系统。
2. 小车开机后 Wi-Fi 正常连接。
3. 网页可访问。
4. SSH 可访问。
5. VNC 可访问。
6. 我们的程序可开机自启或由网页可靠启动。
7. 官方程序不占用摄像头、底盘、I2C 或端口。

## 十、验收清单

1. 小车开机后 Wi-Fi、网页、SSH、VNC 都可用。
2. 官方 App 主程序不再占用硬件资源。
3. 网页显示当前运行模式和硬件连接状态。
4. 网页“开始”能启动真车巡逻。
5. 手动控制按钮能控制真车。
6. 急停始终可用。
7. 云台能初始化到侧向货架视角。
8. 小车能慢速前进、停止、横移、原地旋转。
9. 四路全黑时顺时针转 90 度并继续巡逻。
10. 第一轮不报缺货。
11. 第二轮开始能检测并上报缺货。
12. 能识别货架 AprilTag。
13. 能识别物品 AprilTag、文字和图形。
14. 有颜色时能记录颜色；没有颜色时不报错。
15. 扫描货架和识别物品时播放不同提示音。
16. 缺货时红灯和语音报警。
17. 障碍停车等待 6 秒，未解除则绕行。
18. 绕行过程可被新障碍打断。
19. 网页不显示假的预设固定地图，而显示运行生成的拓扑或轨迹。
20. 异常可在网页人工确认，日志可导出。
