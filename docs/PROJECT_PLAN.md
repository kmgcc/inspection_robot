# 固定仓库场景下的麦克纳姆轮巡逻小车全局规划书

## 结论

本项目现在不再按旧版“黑线巡检 + 回收确认”的思路来做，而是收敛为一个更贴近演示场地、也更能体现小车硬件特点的系统：

**基于固定仓库地图的麦克纳姆轮巡逻、禁区规避、货架侧向扫描、物品异常上报系统。**

它的基本画面是：在一块空旷白色地面上，用黑胶带贴出禁区和边界，放置两列或多列简化货架。小车根据预设仓库地图规划巡逻路线，前方超声波负责行驶方向上的障碍检测，侧向摄像头对着货架扫描货架编号和物品编号；到达货架 A1、A2、B1、B2 等位置时，小车可用麦克纳姆轮横移或原地旋转调整姿态，再进行图像采集和识别。系统把识别结果与货架清单比对，发现缺失、重复、错放、未知物品等异常后，用蜂鸣器和 RGB 灯告警，并把事件上报到本地网页看板。

这里所谓“智能”，不应写成开放世界自动驾驶。课程演示的场地固定、传感器有限，真正可实施的路线是：固定二维地图 + A* 或 Dijkstra 全局路径规划 + 超声波局部避障 + 视觉标签定位 + 货架清单规则比对。LLM 可以放在告警后处理里，用来把多条异常合并成人能读懂的巡检摘要；它不负责实时控制底盘，也不直接决定避障动作。这个边界说清楚，项目反而更可信。

## 一、资料依据

项目使用 RASPBOT V2 麦轮小车。仓库里已经保留了精简后的官方资料，主要依据在 `inspection_robot/docs/RASPBOT-V2_Clean_Docs/`。这些资料足以支撑本项目的演示版本，但也限定了我们不能把目标写得过大。

硬件与系统方面，`17.Docker/5、进入机器人的docker容器/` 说明车上 ROS2 Humble 环境可通过 Docker 使用，并挂载摄像头和 I2C 设备；`15.开发环境搭建/4.远程访问/` 说明远程连接和部署方式。底盘方面，`19.程序源码汇总/04.Car_motion_control/` 下有麦克纳姆轮前后、斜向、横移、旋转的 Jupyter 示例，函数包括 `move_forward`、`move_backward`、`move_left`、`move_right`、`rotate_left`、`rotate_right` 等，这正好对应我们需要的前进巡逻、侧向贴货架扫描和原地调姿。

传感器与告警方面，`03.小车基础教程/06.超声波测距打印/` 和 `19.程序源码汇总/03.Basic_car_course/6.Ultrasonic distance measurement.ipynb` 说明可通过 I2C 读取超声波距离；`05.Comprehensive_gameplay/2.ultrasonic_obstacle_avoidance.ipynb` 已给出基于距离阈值的后退、旋转、前进规则，可作为局部避障动作的参考。`03.小车基础教程/07.四路巡线模块状态打印/` 与 `05.Comprehensive_gameplay/5.car_geofencing.ipynb` 可作为黑胶带边界/禁区检测的参考，但黑胶带在新版方案中不是主巡线路径，而是“不可进入区域”的现场标识或最后防线。蜂鸣器和 RGB 灯分别有 `01.蜂鸣器控制/`、`03.RGB控制/` 以及对应基础课程源码，足以做异常和障碍提示。

视觉方面，`14.ROS2-opencv系列课程/9.AprilTag标签码识别/` 说明官方 AprilTag 使用 TAG36H11，源码使用 `dt_apriltags.Detector` 并发布标签 ID；`07.AI视觉识别/5.二维码识别/` 和 `08.AI视觉交互课程/3.二维码识别控制运动/` 说明二维码识别也是可行路线。我们已经确认现有小方块上的码是 AprilTag TAG36H11，例如样张可被 OpenCV 识别为 `DICT_APRILTAG_36H11`、`ID = 3`。新方案不要求识别任意自然物体，但物品也不能只识别码：AprilTag 负责稳定身份，图像、文字和颜色块同时进入检测范围，用于复核、展示和异常解释。货架识别则以 AprilTag 为主，OCR 读取上方大号货架号作为补充。

## 二、项目目标

必须完成的目标如下：

1. 小车能在固定仓库演示地图中按预设起点、货架点、终点执行巡逻任务。
2. 系统能根据白色地面上的禁区配置生成可行路线，避开黑胶带标记的禁区。
3. 小车前方超声波能检测临时障碍，先停车等待；障碍仍存在时执行规则式绕行，绕行方向优先远离货架。
4. 小车能用 AprilTag TAG36H11 识别当前位置、货架编号和物品编号；货架号同时用 OCR 读取大号字母数字作为补充。
5. 到达货架区域后，小车能调整姿态或横移，让侧向摄像头面向货架并完成扫描。
6. 系统能把某一货架的实际识别结果与预设清单比对，发现物品缺失、重复、错放和未知标签；物品识别结果应记录 AprilTag、颜色、文字/OCR 和图像类别中的可用证据。
7. 异常发生时，小车能通过蜂鸣器、RGB 灯和网页看板告警，并记录事件。
8. 本地网页看板能显示地图状态、当前目标点、当前货架、最近识别结果、障碍状态、异常列表和导出日志。

加分目标可以做，但不进入主链路：黑胶带禁区的自动视觉更新；LLM 生成巡检摘要和处置建议；货架扫描图片或关键帧保存；多个巡检任务模板；简单语音或按钮触发任务。它们都应建立在主链路跑通之后。

明确不做的内容也要写清楚：

1. 不做 SLAM，不要求未知环境建图。
2. 不做开放场景自由自动驾驶。
3. 不训练大型深度学习模型，不承诺识别任意物体。
4. 不做机械臂抓取或自动搬运。
5. 不让 LLM 直接控制底盘、电机或避障动作。
6. 不把黑胶带当作主巡线路径；黑胶带在新版方案里表示禁区、边界或局部保护线。

## 三、总体架构

系统采用“固定地图规划 + 车端执行 + 视觉识别 + 核心规则 + 网页看板”的结构。它仍然是小系统，但每一段链路都能在答辩时讲清楚。

```text
固定仓库地图 / 货架清单 / 禁区配置
        |
        v
路径规划模块（A* / Dijkstra）
        |
        v
运动执行模块 ---- 麦克纳姆轮底盘
        |          |
        |          +-- 前进 / 后退 / 横移 / 原地旋转
        |
        +-- 超声波局部避障
        +-- 黑胶带禁区检测兜底
        |
        v
侧向视觉识别模块
        |
        +-- AprilTag TAG36H11 / QR / OCR / 颜色 / 图像
        +-- 货架编号 / 物品编号 / 识别证据
        |
        v
异常规则模块
        |
        +-- 缺失 / 重复 / 错放 / 未知
        |
        v
告警与网页看板
```

主程序仍建议用 Python 实现。硬件侧优先复用官方 `McLumk_Wheel_Sports` 和 `Raspbot_Lib` 示例；视觉侧优先使用 `cv2.VideoCapture(0)` + `dt_apriltags` 或二维码识别；网页侧继续使用当前 Flask 看板和 `/api/status` 轮询。当前仓库已经有 0.1 最小软件闭环，下一步不是推倒重写，而是在共享契约上补货架、地图、路径和扫描字段。

## 四、场景设计

演示场地建议保持固定而小。地面用白色板材或白纸铺开，黑胶带贴出禁区，如“不可进入区”“货架安全距离线”“通道边界”。黑胶带在地图中提前配置为禁区；底部四路红外传感器则作为现场兜底，任意一路检测到黑线时先停车、后退，再按地图选择远离货架的一侧绕开。货架按水平一列摆放，例如左侧为 A1、A2，另一位置为 B1、B2。每个货架正面或旁边放置一个小柱子/纸牌，版式固定为上方大号货架号、下方 AprilTag、底部数字脚注；机器优先读 AprilTag，OCR 只作为补充和复核。每个物品贴物品标签，标签应同时包含 AprilTag、颜色块、简化图像和文字名称。物品可以是小瓶子、纸盒、纸团、积木块等，系统身份仍以标签和配置表为准，颜色和图像用于增加检测维度，而不把自然物体识别作为唯一依据。

摄像头朝机身侧向，而不是朝前方。这样小车沿通道移动时，前方超声波始终在行驶方向上看障碍，侧向摄像头则持续面向货架；需要换货架列时，小车再用麦克纳姆轮横移或原地旋转调整姿态。这个设计应写成项目亮点，因为它不是为了炫技，而是把麦克纳姆轮的横移能力用在了货架巡检这种真实需求上。

路径规划采用固定栅格地图。地图可以用一个 JSON 文件表示：0 表示可通行，1 表示禁区，S 表示起点，A1/A2/B1/B2 表示货架巡检点。全局路径用 A* 或 Dijkstra 生成，运行时按路径点逐段执行。若前方检测到障碍，先停车等待 1-3 秒；若障碍解除，恢复原路径；若障碍仍在，按照当前地图临时把该格或前方小区域标记为不可通行，重新规划一条短路径，或执行固定绕行动作。演示场地很小，所以重新规划不必复杂，关键是规则明确、视频能复现。

## 五、标签、ID 与打印素材约定

AprilTag 统一使用 TAG36H11。ID 不按“看到什么就随便贴什么”处理，而是分段管理：物品使用 `1-50`，货架使用 `101-120`，定位点预留 `201-220`，特殊区域或禁区提示预留 `301-320`。这样即使物品和货架都使用同一种 AprilTag，核心系统也能通过配置表判断它是货架、物品还是定位点，不会把 `ID=3` 同时解释成货架和物品。

货架标签采用“上字下码”的固定版式。上方是人眼可读、也便于 OCR 的 `A1`、`B2` 等大号货架号；下方是 AprilTag；码下方保留 `TAG ID: 101` 这类脚注，方便贴纸时人工核对。物品标签采用“颜色块 + 简化图像 + 文字 + AprilTag”的版式，既能支持稳定 ID 识别，也能让颜色识别、OCR 和图像分类参与检测。已经生成的素材位于仓库根目录 `打印素材_AprilTag/`，其中 `item_labels_A4.pdf` 和 `shelf_labels_A4.pdf` 可直接打印，`manifest.csv` 记录 ID、货架号、颜色和物品类型的对应关系。

## 六、功能模块

**1. 基准契约模块**

这是四个子计划共同依赖的底座。`docs/api_contract.md` 继续规定 `/api/status`、事件结构、状态枚举和 `InspectionStore` 对外方法。旧版字段如 `current_zone`、`current_tag`、`current_item` 不删除；新版在此基础上增加 `current_shelf`、`current_target`、`pose`、`path`、`forbidden_zones`、`scan` 等字段。这样 1.1 的硬件同学、2.1 的核心算法同学、3.1 的看板同学可以并行做，不必互相等待。

**2. 运动与感知模块**

该模块负责真实小车输入输出，不判断业务异常。它需要提供低速稳定的运动原语，包括前进、后退、左横移、右横移、左旋、右旋、停止；读取超声波距离；读取黑胶带/巡线传感器作为禁区兜底；读取侧向摄像头中的标签 ID；把这些原始观察转发给核心状态。它的成功标准不是“跑得快”，而是在演示地图上每次动作都可控。

**3. 路径规划与状态机模块**

该模块不直接碰硬件。它读取固定地图、禁区、货架点和任务列表，生成巡检路线；运行时根据障碍事件和扫描完成事件推进状态机。状态建议包括 `IDLE`、`PLANNING`、`MOVING`、`OBSTACLE_WAIT`、`REROUTING`、`ALIGNING_SHELF`、`SCANNING_SHELF`、`ANALYZING`、`WAIT_CONFIRM`、`FINISHED`、`STOPPED`。旧的 `PATROL` 可以兼容为 `MOVING` 或 `SCANNING` 的上层概念。

**4. 货架识别与异常规则模块**

该模块把识别结果落到“货架编号 + 物品清单”上。货架以 AprilTag ID 作为主键，OCR 货架号作为补充校验；物品以 AprilTag ID 作为主键，同时记录颜色、文字和图像识别证据。示例配置可以写成：

```json
{
  "A1": {
    "expected_items": ["item_01", "item_02", "item_03"]
  },
  "A2": {
    "expected_items": ["item_04", "item_05"]
  }
}
```

当 A1 实际识别到 `item_01, item_02, item_02` 时，系统应生成两类异常：`item_03` 缺失，`item_02` 重复。若在 A1 识别到属于 A2 清单的物品，则生成错放事件。若识别到不在任何清单里的标签，则生成未知物品事件。若 AprilTag 与 OCR、颜色或图像结果冲突，例如 `item_02` 的颜色应为黄色但实际颜色检测为蓝色，应记录为识别证据不一致，优先进入人工复核。算法不必神秘，本质就是稳定识别和清单比对；答辩时讲得越清楚，越像工程项目。

**5. 告警、看板与 LLM 后处理模块**

蜂鸣器和 RGB 灯负责现场提示：正常扫描绿色短闪；普通异常黄色；高优先级异常红色并蜂鸣；障碍等待蓝色或紫色。网页看板负责展示地图、路径、当前货架、识别结果、异常事件、确认按钮和 CSV 导出。LLM 只作为加分项：把多条异常整理成自然语言报告，例如“本轮巡检 A1 缺少 item_03，A2 出现 item_07 重复，建议人工复核 A1/A2 货架”。即使 LLM 不接入，主系统也必须能独立完成巡逻、识别和上报。

- `docs/plans/1.1-plan-robot-io.md`：队友 1。负责小车硬件、麦轮运动、超声波、黑胶带传感器、侧向视觉和 runtime 接线。
- `docs/plans/2.1-plan-core-contract.md`：队友 2。负责固定地图、路径规划、货架清单、异常规则、状态机、持久化和测试。
- `docs/plans/3.1-plan-dashboard-demo.md`：队友 3。负责网页看板、模拟演示、部署脚本、运行手册和答辩证据。

这四份文档的接口关系必须保持简单：1.1 只上报观察和执行运动，不写业务规则；2.1 只做规划和判定，不碰真实硬件；3.1 只消费 `/api/status` 和事件，不复制规则；0.1 负责把字段说清楚。这样三个人可以并行推进，最后在 Flask 看板和 `InspectionStore` 上合流。

## 八、推进顺序

第一天先完成文档对齐和硬件冒烟。确认前后移动、横移、旋转、超声波、四路红外黑胶带检测、RGB、蜂鸣器、AprilTag TAG36H11、OCR、颜色识别能跑；同时把白地、黑胶带禁区、上字下码货架标签、多模态物品标签布好。此时不追求自动路线，只要证明各模块单独可用。

第二天完成固定地图和路径规划的纯软件版本。用 JSON 写出地图、禁区和货架点，在本地用 A* 或 Dijkstra 生成路径，并把路径显示到看板或终端。没有小车也要能跑通模拟任务：从起点到 A1，再到 A2、B1、B2，最后回终点。

第三天接入麦克纳姆轮运动原语和超声波。小车按简单路径点低速运行，遇到障碍先停，再恢复；如果时间允许，做固定绕行动作或短路径重规划。绕行方向要按地图配置优先远离货架，以免小车往货架方向挤。

第四天接入侧向视觉扫描。到达货架点后，小车减速或停车，调整姿态，让摄像头对准货架，识别货架编号和物品编号。识别不稳时使用多帧投票，宁可慢一点，也不要让事件乱跳。

第五天完成货架清单比对、异常事件、告警和看板。重点演示缺失、重复、错放、未知物品四类异常。网页上要能看到当前路径、当前货架、异常列表、确认处理和 CSV 导出。

第六天做联调和答辩材料。准备真车演示和软件兜底两套流程。真车流程展示路径规划、禁区规避、障碍处理、侧向扫描和异常上报；软件兜底流程用按钮模拟同样事件，避免现场网络或硬件失败导致无法讲清系统。

## 九、演示流程

标准真车演示建议控制在 2-3 分钟：

1. 打开看板，展示地图、禁区和任务列表。
2. 点击开始巡逻，小车从起点出发，路径避开黑胶带禁区。
3. 在前方放置小瓶子或纸团，小车超声波检测到障碍并停车等待。
4. 障碍移开后恢复；或障碍不移开时，小车执行绕行或短路径重规划。
5. 到达 A1 货架，小车横移或旋转，使侧向摄像头对准货架。
6. 识别 A1 货架标签和物品标签，页面展示 AprilTag、OCR、颜色和图像证据，生成正常记录。
7. 在 A2 或 B1 人为制造缺失、重复或错放，系统生成异常。
8. RGB/蜂鸣器告警，看板显示异常和货架编号。
9. 点击人工确认处理，事件状态关闭，导出 CSV。

软件兜底演示必须保留同样叙事：开始任务、规划路径、扫描 A1、模拟障碍、模拟 A2 缺失/重复、确认处理、导出日志。它不替代真车，但能保证答辩时主链路不丢。

## 十、风险与兜底

最大风险不是算法，而是现场不稳定。AprilTag 识别不稳时，增大标签、固定距离、停车识别、多帧投票；OCR 识别不稳时，把它降为复核字段，不让它推翻 AprilTag 主身份；颜色受光照影响时，先做白地/颜色块阈值标定，并把颜色结果作为证据而非唯一身份；超声波误报时，连续多次低于阈值再停车；麦轮运动漂移时，缩短路径点距离，降低速度，并用人工摆正或视觉标签重新校正；黑胶带传感器不稳定时，把禁区主要放进固定地图，传感器只作为兜底停机信号；网络失败时，用终端日志和本地录屏补展示。

另一个风险是目标写得太像真正仓库机器人。我们要反复强调：这是课程演示场景，是固定小地图，不做 SLAM，不处理开放环境。固定场景不是短板，而是工程取舍；在传感器有限的条件下，能稳定完成路径规划、避障、货架扫描和异常上报，已经足够构成完整系统。

## 十一、验收清单

- [ ] 白色地面、黑胶带禁区、至少两列货架布置完成。
- [ ] 小车能前进、后退、横移、旋转、停止。
- [ ] 超声波能触发障碍等待，解除后恢复。
- [ ] 底部四路红外传感器能识别黑胶带禁区边界，并触发停车/后退/避让或禁区等待。
- [ ] 固定地图能生成避开禁区的巡检路径。
- [ ] 至少识别 4 个 AprilTag TAG36H11 标签，且能区分物品 ID 和货架 ID 范围。
- [ ] 货架标签采用上方货架号、下方 AprilTag 的版式，OCR 货架号可作为补充识别。
- [ ] 物品标签包含 AprilTag、颜色块、图像和文字，系统能记录至少两类识别证据。
- [ ] 侧向摄像头能完成至少一列货架扫描。
- [ ] 能检测缺失、重复、错放、未知物品中的至少三类。
- [ ] 蜂鸣器和 RGB 灯能区分正常、障碍、异常。
- [ ] 看板能显示地图/路径/货架/异常/确认/导出。
- [ ] 有真车演示视频、软件兜底录屏和 CSV 样例。

最终答辩题目建议使用：

**固定仓库场景下的麦克纳姆轮智能巡逻与货架异常检测系统**

这个名字把场景、底盘优势和功能边界都写进去了。它不夸口做通用自动驾驶，也不退回简单巡线；它承认场景固定，同时把路径规划、禁区规避、侧向扫描和异常上报这几件事串成一条能落地的工程链路。
