# 巡逻小车直行左偏问题 —— 代码审查报告

> 审查对象：`src/inspection_robot/`（RASPBOT V2 麦轮巡逻小车）
> 症状：直行时持续向左偏，且越偏越厉害；巡逻时时不时顿挫后突然转弯；偶发单轮转动、其余三轮卡死。
> 审查日期：2026-07-05　审查范围：行车轨迹 / 航向保持 / 匀速巡航 / 转向相关代码
> 结论：**问题高度集中在昨天新增的"直行航向保持（heading hold）+ 匀速巡航"改动上，而不是原有的路线/转向逻辑。**

---

## 一、结论速览（先看这里）

你的三个症状不是三个独立的 bug，而是**同一套新代码（昨天的航向保持 + 匀速巡航）**在不同侧面上的表现：

| 你观察到的现象 | 直接根因 | 代码位置 |
|---|---|---|
| ① 直行"一直往左偏，越来越厉害" | **航向保持的纠偏方向极可能是"正反馈"**（越纠越歪，发散） | `heading_hold.py:63`、`mpu6050.py:848` |
| ① + ② 陀螺仪纠偏"无效/帮倒忙" | **漏积分器 + 粗采样**：极小的零漂就会产生一个恒定的"幻影航向"，让小车被持续朝一个方向脉冲纠偏 | `mpu6050.py:153-172` |
| ② 时不时停顿，然后突然转弯 | **纠偏动作是"停车→原地旋转脉冲→停车→再前进"**，本质上打断了匀速直行 | `heading_hold.py:67-69`、`runtime.py:622` |
| ③ 单轮转、其余三轮卡死 | **巡航速度 14 太低**，接近麦轮静摩擦启动阈值；反复"停-转-走"让电机反复从零启动，部分轮子憋不动 | `runtime.py:119`（`cruise_speed=14`） |

**最可疑的头号根因：航向保持的纠偏极性（方向）从来没有在真车上标定过，而且现有的闭环转向测试和单元测试都无法暴露这个错误。** 一旦极性反了，"陀螺仪纠偏"就从"纠正"变成"放大"，完全符合"越走越左、越偏越狠"的发散现象。

## 三、逐条根因分析

### 根因 1【头号嫌疑】航向保持纠偏方向未标定，极可能是正反馈 → 越走越偏

**代码路径：** `runtime.py:_apply_heading_hold()` → `heading_hold.py:apply_heading_hold()`

核心决策在 `heading_hold.py:45-69`：

```python
deviation = float(updater())          # 来自 StraightHeadingGuard.update()，是"积分出的航向角"
...
effective = deviation + damping * rate
...
turn_right = effective > 0.0          # ← 航向角为正就右转脉冲，为负就左转脉冲
if settings.invert:
    turn_right = not turn_right
mover = motion_adapter.rotate_right_slow if turn_right else motion_adapter.rotate_left_slow
mover(speed=speed, duration_seconds=pulse_seconds)
```

而 `deviation`（航向角）的符号由 `StraightHeadingGuard.update()`（`mpu6050.py:153`）决定：

```python
rate = float(rates.get(self.yaw_axis, 0.0)) * self.yaw_sign   # yaw_sign 默认 = -1
...
self.heading_degrees = _wrap_degrees(self.heading_degrees * decay + rate * elapsed)
```

**符号链推导（默认参数：yaw_axis=z, yaw_sign=-1）：**

1. MPU6050 的 Z 轴陀螺，按右手定则，小车**向左（逆时针 CCW）偏航时 gyro_z 为正**。
2. `rate = gyro_z × yaw_sign = 正 × (-1) = 负` → 航向角 `heading_degrees` 累积为**负**。
3. `deviation < 0` → `effective < 0` → `turn_right = False` → 执行 **`rotate_left`（继续左转）**。

也就是说：**小车往左偏 → 系统命令它继续左转 → 偏得更多 → 再左转……正反馈发散。** 这与"一直往左偏、而且越来越厉害"完全吻合。

> ⚠️ 需要现场确认的前提：上面推导依赖 (a) MPU6050 的实际安装朝向（Z 轴朝上还是朝下），(b) 闭源库 `rotate_left/rotate_right` 的实际转向定义。这两点在仓库里都无法确认（驱动库闭源，只在车上）。**但无论正负，结论都是"这个方向从未被验证过"，而症状恰好是极性反了时会出现的样子。**

**为什么已有的"陀螺仪闭环转向 turn_90"正常，却不能证明这里也对？**

因为 `turn_90` 测量转角时取的是**绝对值**（`mpu6050.py:_turn_measurement_from_axes` 第 677 行 `abs(float(...))`）——它只"命令一个方向 + 测量转了多少度（不看正负）"，从不校验"陀螺仪的正号是否对应命令的方向"。所以 `turn_90` 能收敛，**完全不能说明** heading hold 里"航向角符号 → 纠偏方向"的映射是对的。这是两套独立的符号约定，只有 heading hold 这套是新的、且未标定。

**为什么单元测试全绿也没用？**

`tests/test_runtime_cruise.py:213-227` 的 `test_heading_hold_pulses_when_deviation_worsening` 用的是 `FakeGuard(deviation=6.0, rate=10.0)`，直接把"航向角"喂进去，断言"正 → rotate_right"。它验证的只是**代码内部自洽**（正 deviation 就右转），**完全绕过了真实陀螺仪的符号、安装朝向、电机转向**。真实链条里"小车向左偏 → deviation 是正还是负"这一环，没有任何测试覆盖。这就是"测试全过但上车帮倒忙"的原因。

### 根因 2　纠偏是"停车→原地旋转→停车→再前进"，打断匀速 → 顿挫后突然转弯（对应症状①的"停顿"）

匀速巡航每个 tick（`runtime.py:_drive_cruise_step`，第 613 行）第一步就是：

```python
self._apply_heading_hold()      # 第 622 行
```

而 `apply_heading_hold` 一旦判定要纠偏，动作是（`heading_hold.py:67-69`）：

```python
mover(speed=speed, duration_seconds=pulse_seconds)  # 原地旋转脉冲（rotate_left/right）
motion_adapter.stop()                               # 然后停车
```

注意 `rotate_left/right` 在麦轮上是**原地自转**（左右两侧轮子反向转）。所以真实时序是：

```
…前进(keep_running 不停) → 需要纠偏 → 原地自转一下 → 停车 → 回到主循环 → 再 move_forward …
```

从旁观者看，就是**"匀速走着走着，突然顿一下、几个轮子往反方向动了一下（自转），然后又走"**——正是你说的"时不时停顿，然后突然有几个轮子不动/开始转弯"。

匀速巡航（`7f7466e`）的初衷本是"持续前进 + 微脉冲纠偏、不要停-走-停"，但 `apply_heading_hold` 里每次脉冲后都硬 `stop()`，且脉冲用的是"原地自转"而非"边走边微调"，等于把刚去掉的顿挫又加了回来。

> 补充：在**旧的短步巡逻模式**（`SMOOTH_CRUISE_ENABLED=0`）里更密集——`_forward_step`（`runtime.py:1306`）传 `heading_hold=watch_boundary=True`，于是 `_run_timed_motion` 会在**每个 0.02s 的运动分片前都调一次** `_apply_heading_hold`（`runtime.py:805`）。一个 0.18s 的前进步里最多插 9 次原地自转脉冲，顿挫和发散都会被放大。

### 根因 3　漏积分器 + 粗采样：极小零漂就产生恒定"幻影航向"，被持续单向纠偏（对应症状②，也放大主症状）

`StraightHeadingGuard.update()`（`mpu6050.py:153-172`）是一个**漏积分器（leaky integrator）**：

```python
leak = leak_per_second * elapsed          # leak_per_second 默认 0.08
decay = 1.0 - leak
heading = heading * decay + rate * elapsed
```

漏积分器的稳态解是 `heading_稳态 ≈ rate / leak_per_second`。代入默认参数：

- 死区 `deadband_dps` 默认 **0.25**（`mpu6050.py:856`）：只有 |rate| < 0.25 dps 才当作 0。
- 漏率 `leak_per_second` 默认 **0.08**（`mpu6050.py:863`）。
- 纠偏触发阈值 `tolerance_degrees` 默认 **3.0°**（`heading_hold.py:11`）。

于是：**任何一个刚好越过死区（比如 0.3 dps）的残余零漂，稳态航向 = 0.3 / 0.08 ≈ 3.75°，直接超过 3° 触发阈值。** 也就是说，只要陀螺仪有一丁点没被标定掉的零偏（这是常态，尤其温度漂移），漏积分器就会把"幻影航向"稳定停在 ~3–6°，于是系统会**恒定地、朝同一个方向、不停地发纠偏脉冲**，哪怕小车其实在走直线。

这三个参数（死区 0.25 / 漏率 0.08 / 阈值 3.0）之间是**互相打架**的：死区放行的最小零漂，其稳态幻影航向恰好压着触发阈值。结果就是"陀螺仪纠偏"要么无效、要么持续把车往一个方向拱——正好对应"即便用陀螺仪调整也依然无效/帮倒忙"。

零漂重标定（ZUPT，`_zupt_recalibrate`）只在**停车时**触发（巡航开始、障碍停车、列端转向后），直行途中零漂随温度继续漂，无人纠正。

另外，`update()` 每次只在被调用的**那一瞬间采一个陀螺样本**，再乘以"距上次调用的时间差"当作整段时间的角速度。而它是**每个控制循环才调一次**（巡航约每 0.05s，若中间有扫描/边界停车，`elapsed` 会被拉到上限 0.25s）。这种"单点采样 × 粗时间步"的积分本身就很不准，进一步让纠偏时灵时不灵。

### 根因 4　巡航速度 14 太低，接近麦轮静摩擦启动阈值 → 单轮转、三轮卡（对应症状③）

`RobotRuntimeConfig.cruise_speed` 默认 **14**（`runtime.py:119`，满量程 100）。麦克纳姆轮 + 减速电机在这么低的 PWM 下，**动摩擦勉强能维持、但静摩擦（从零启动）往往顶不动**。

而根因 2 造成的"停车→原地自转→停车→再前进"churn，让四个电机**反复从零静止重新启动**。在 PWM≈14 这个档位，静摩擦力矩大于电机力矩时，就会出现**只有附着/装配最好的那一个轮子转起来、其余三个憋住不动**的现象——正是你说的"某一个轮子转动，而其他三个轮子卡住，导致航向偏离"。每次这种不对称启动都会给车身一个随机的偏航冲量，叠加到左偏上。

`ROBOT_COMMAND_REPEAT`（默认 2，`motion.py:15`）会把每条电机指令连发 2 次，再叠加"stop→rotate→stop→forward"的高频指令切换，在 I2C 上短时间密集下发不同的轮子方向指令，也可能出现部分指令被覆盖/丢失，加剧"只有一个轮子动"的表现。

## 四、次要 / 放大因素

- **纠偏脉冲自身的旋转没有被反馈测量**：`apply_heading_hold` 在 `guard.update()` 之后才发脉冲，脉冲期间车身的转动不会被下一次 `update` 之前的积分捕捉到，等于开环打一拳、下一拍再盲测，容易过冲/欠冲。
- **`keep_running=True` 与脉冲 `stop()` 混用**：匀速路径靠"最后一片不 stop"来保持连续，但 heading hold 每次都 `stop()`，两种意图在同一个 tick 里互相拆台。
- **短步模式纠偏频率过密**（见根因 2 补充），如果你临时切回 `SMOOTH_CRUISE_ENABLED=0` 想规避匀速问题，反而会撞上更密集的脉冲。

## 五、如何验证 + 快速止血（建议按顺序做）

### 第 1 步：先隔离，确认是不是 heading hold 在帮倒忙
上车加环境变量 **关闭直行陀螺仪纠偏**，其它不动：

```bash
HEADING_HOLD_ENABLED=0 ./scripts/run_on_car.sh
```

- 若关掉后**"越走越偏"的发散消失了**（可能还有轻微机械性固定偏移，但不再自我放大）→ **确诊根因 1/2/3**，问题就在航向保持闭环。
- 若关掉后**照样越偏越狠** → 说明还有机械/电机层问题（比如某个电机/轮子本身偏弱），再查根因 4 与硬件。

### 第 2 步：如果确诊是纠偏极性反了，验证并翻转
```bash
# 二选一即可先验证极性：
HEADING_HOLD_INVERT=1 ./scripts/run_on_car.sh        # 只翻纠偏方向
# 或
MPU6050_YAW_SIGN=1 ./scripts/run_on_car.sh           # 翻陀螺仪航向符号（会同时影响航向显示）
```
让小车直行，观察是否变成"能自动拉回直线"。**注意：这两个不要同时开**（会互相抵消）。同时用看板/日志看 `heading_hold_correction` 事件里的 `deviation_degrees` 和 `direction`：手动把车头掰向左，看 deviation 是正是负、系统发的是 left 还是 right，就能一锤定音地定极性。

### 第 3 步：缓解"低速憋轮"
```bash
CRUISE_SPEED=22 ./scripts/run_on_car.sh   # 抬到不憋轮的最低稳定速度（22 是原短步默认速）
```
先把速度抬到四个轮子都能从静止顺利起步的档位，观察症状③是否消失，再往下调找"匀速但不憋"的临界点。

### 第 4 步（若还想留纠偏）：把"幻影航向"参数调开
把死区/漏率/阈值调得不再互相打架，避免微零漂就触发：
```bash
HEADING_HOLD_TOLERANCE_DEG=6   MPU6050_YAW_DEADBAND_DPS=0.6   MPU6050_YAW_LEAK_PER_SECOND=0.3 ./scripts/run_on_car.sh
```

> 以上都是**环境变量级的快速试探**，不改代码、可随时回退，适合现场逐个坐实根因。

## 六、根本修复方向（确诊后再动代码）

1. **标定并固化纠偏极性**：上车用第 2 步的方法确定正确方向后，把 `MPU6050_YAW_SIGN` / `HEADING_HOLD_INVERT` 的正确值写进 `run_on_car.sh` 或 `calibration.json`，并在代码里加一段**开机自检**（命令小车左转一点、读陀螺仪、断言符号），把"物理极性"纳入可测试范围，避免再次靠肉眼。
2. **把"原地自转脉冲纠偏"改成真正的"边走边微调"**：直行时不要 `stop()` + 原地自转，而是给左右两侧一个小的速度差（差速），保持前进的同时缓慢回正；或至少去掉脉冲后的硬 `stop()`。这需要先上车标定单轮/差速符号（记忆里已注明"要用差速必须先标定每轮正转符号"）。
3. **纠偏改成真正的闭环反馈**：脉冲后重新采样航向，用比例项收敛，而不是开环打一拍盲测下一拍；提高采样频率或用固定 dt。
4. **参数体检**：让 `deadband / leak / tolerance` 之间留出安全裕度（当前配置下"死区外最小零漂的稳态航向"≈触发阈值，属于误设计）。
5. **确认低速可启动性**：把 `cruise_speed` 定在"四轮都能从静止起步"的下限之上，或加"起步给一个短促高 PWM 冲量再落到巡航速度"的起步补偿。

## 需要你确认的信息（会直接影响修复走向）

1. 现在上车实际用的是**匀速巡航（`SMOOTH_CRUISE_ENABLED=1`）**还是短步（=0）？两者都有问题但表现细节不同。
2. MPU6050 在车上的**安装朝向**（芯片正面朝上还是朝下 / 有没有被翻转贴装）——决定 Z 轴符号，也就决定极性推导是否成立。
3. 关掉 `HEADING_HOLD_ENABLED=0` 之后，小车直行**是否还发散**？这一条能立刻把"闭环 bug"和"机械/电机偏弱"分开。
4. 症状③（单轮转）是**只在纠偏/起步瞬间**出现，还是**匀速平飞时也会**出现？前者指向根因 4，后者要另查驱动/电机接线。
