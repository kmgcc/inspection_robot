# 真实需求基准文档

**状态：** 当前最重要的需求文档。  
**用途：** 记录真实小车场地、真实运行逻辑、网页控制、运维要求和当前问题。  
**优先级：** 如果本文与 `PROJECT_PLAN.md`、`docs/api_contract.md`、`docs/plans/0.1-3.1` 或 README 中的旧描述冲突，以本文为准。后续所有计划、代码和测试都应向本文对齐。

## 一、项目真实定位

本项目不是循线小车，也不是让小车沿黑线行驶。真实目标是：

**在固定货架通道中，让 RASPBOT V2 麦克纳姆轮小车自动往返巡逻，侧向扫描货架与物品，发现缺货等异常后用灯光、声音、网页看板进行提示和上报。**

小车的基本运行画面是：

1. 小车放在初始位置后，开机或点击网页“开始”即可慢速向前巡逻。
2. 小车沿货架列之间的通道行驶，摄像头朝向机身侧面，而不是朝向正前方。
3. 侧向摄像头在经过货架时识别货架 AprilTag，并识别货架上的物品。
4. 小车在货架尽头遇到横向黑胶带时，四路底部传感器同时检测到黑色，系统判断到达列端或禁区触发区。
5. 触发后小车顺时针原地旋转 90 度，然后继续前进。
6. 通过每列货架尽头的“工字形”黑胶带布置，小车可以在 A 列、B 列等货架之间往返循环巡逻。
7. 第一轮巡逻只用于建立运行状态，不做缺货判断；从第二轮开始，实时检测并上报缺货等异常。

项目边界仍然保持清楚：

1. 不做 SLAM。
2. 不做未知环境自由自动驾驶。
3. 不让 LLM 控制底盘。
4. 不做机械臂搬运。
5. 不把黑胶带当作主行驶路线。
6. 地图不是预先人工画死的栅格路径；网页显示的地图应来自运行中检测到的货架、物品、边界和异常。

## 二、场地与货架布局

### 2.1 真实货架排列

当前已明确：

1. A 列货架按行驶方向依次为 `A1`、`A2`、`A3`、`A4`。
2. 每个货架点都有一堆货物。
3. 小车沿货架列之间通道前进，侧向摄像头对准货架。
4. 到达 A 列货架尽头时，小车会遇到横向黑胶带。
5. 横向黑胶带不是巡线轨道，而是列端/禁区触发标记。
6. 每列货架尽头的垂直方向安装“工字形”黑胶带，用于让小车在列端转弯，并在另一头再次转弯。
7. 通过这种方式，小车可以从 A 列切换到 B 列，摄像头转而对准 B 列货架。

仍需确认：

1. B 列是否也为 `B1`、`B2`、`B3`、`B4`。
2. 后续是否还有 C 列、D 列。
3. 每一列的货架编号方向是否都按小车实际经过顺序编号。
4. 每个货架的物品清单是否固定，还是允许现场调整后重新配置。

### 2.2 黑胶带语义

黑胶带有两类语义，必须区分：

1. **四路全黑：** 到达货架尽头、列端或禁区触发区。此时执行端点动作。
2. **局部压黑或传感器异常：** 安全兜底信号。此时应进入保护逻辑，不应误当作正常巡线。

端点动作的当前要求：

1. 停车。
2. 顺时针原地旋转 90 度。
3. 继续慢速前进。

注意：旧文档中“任意一路压黑就后退、绕开、按地图规避”的描述不再是主逻辑。真实主逻辑是：四路同时检测到黑胶带时，说明到达货架尽头，需要顺时针转向继续巡逻。

## 三、巡逻运行逻辑

### 3.1 启动行为

小车开机时由人工放在初始位置。启动后应满足：

1. 云台先初始化，使摄像头朝向机身侧面货架。
2. 小车进入待命或自动巡逻状态。
3. 如果设置为开机自启，系统应在不需要人工 SSH 操作的情况下启动网页服务和机器人 runtime。
4. 如果使用网页启动，则点击“开始”后小车应立即慢速向前行驶。
5. 开始后网页应明确显示当前状态为巡逻中、移动中或扫描中。

当前遇到的问题：

1. 点击网页“开始”后机器人没有动。
2. 之前小车开机半天不动，说明 runtime 没有真正接管底盘，或没有以正确模式启动。
3. 手动控制按钮提示必须进入 robot mode，导致前进、转向、云台等功能无法直接使用。

### 3.2 循环检测

巡逻必须是持续循环，而不是跑完一条固定路径就结束。

循环逻辑：

1. 第 1 轮：只运行、校准、识别经过的货架，不上报缺货。
2. 第 2 轮及之后：开始按货架清单检查物品，发现缺失、重复、错放、未知物品或证据冲突时实时上报。
3. 每次经过货架时，如果识别到货架 AprilTag，更新当前货架。
4. 每次识别到物品时，记录该物品属于当前货架的一次观察。
5. 每轮结束时，系统可对这一轮看到的物品集合与预期清单进行比对。
6. 如果需要实时性，也可以在离开某个货架区域后立即对该货架做局部比对。

### 3.3 行驶规范

小车应尽量保持行驶方向为正向，也就是通过绕行和转向动作恢复原行驶方向，而不是长期倒车或以错误朝向巡逻。

原则：

1. 正常巡逻时低速前进。
2. 到货架尽头时按端点逻辑转向。
3. 遇到障碍或禁区需要绕行时，优先向外侧绕行，避免向货架方向挤压。
4. 绕行结束后恢复原方向，继续向前巡逻。
5. 不应因为临时避障导致摄像头长期背向货架或行驶方向混乱。

## 四、识别需求

### 4.1 货架识别

货架通过 AprilTag 识别。

要求：

1. 每个货架必须贴有货架 AprilTag。
2. AprilTag 是货架身份的主来源。
3. 货架上的大号文字，如 `A1`、`A2`，可以作为 OCR 辅助校验，但不能替代 AprilTag 主身份。
4. 识别到货架后，系统应更新当前货架编号。
5. 识别到货架时，应播放货架扫描提示音。

### 4.2 物品识别

物品识别需要支持三种形式：

1. 图形识别。
2. 文字识别。
3. AprilTag 识别。

其中 AprilTag 可以作为稳定身份来源；图形和文字可以作为补充证据。后续如果实际物品没有 AprilTag，也应允许通过图形和文字形成可用观察，但置信度和异常判断策略要更保守。

颜色识别要求：

1. 如果物品有颜色标识，应识别颜色。
2. 支持至少蓝、绿、黄、红等常见颜色。
3. 颜色用于物品价值、属性或复核判断。
4. 如果没有颜色，系统不能报错。
5. 缺少颜色时，应将颜色字段记为未知、空值或未提供，而不是当作异常。

识别到具体物品时，应播放另一个物品提示音，与货架扫描提示音区分。

### 4.3 异常判断

从第二轮巡逻开始，系统需要判断：

1. `missing_item`：预期存在但当前轮未识别到。
2. `duplicate_item`：同一货架上识别到重复物品。
3. `wrong_shelf`：物品出现在错误货架。
4. `unknown_item`：识别到不在配置表中的物品。
5. `evidence_mismatch`：AprilTag、图形、文字、颜色之间存在冲突。

缺货异常的现场反馈：

1. 亮红色 LED。
2. 发出语音警报。
3. 语音内容应能表达类似“检测到某某物品缺失”。
4. 网页看板显示缺失物品、所在货架、时间、事件状态。
5. 事件进入待确认或待处理状态。

## 五、音频与灯光提示

### 5.1 音频提示类型

至少需要三类音频/语音能力：

1. **货架提示音：** 巡检过程中扫描到货架时播放。
2. **物品提示音：** 识别到具体物品时播放。
3. **异常语音警报：** 识别到缺货、错放等需要处理的异常时播放。

### 5.2 异常语音

缺货语音示例：

```text
检测到 A2 货架 item_04 缺失，请及时补货。
```

语音系统第一阶段可以只预留接口，第二阶段接入固定音频文件，第三阶段再做动态 TTS。

要求：

1. 没有音频文件时，不应影响小车巡逻。
2. 音频播放失败时，应记录事件或状态，但不能让主程序崩溃。
3. 网页应提供测试播放或预留播报按钮。
4. 小车本体音频输出优先通过树莓派连接的蓝牙音箱或本地音频设备播放，而不是浏览器所在电脑播放。

### 5.3 灯光提示

建议状态：

1. 正常巡逻：绿色或低亮度常规状态。
2. 扫描货架：绿色短闪。
3. 识别物品：蓝色或白色短闪。
4. 障碍等待：蓝色或紫色。
5. 缺货、错放、高优先级异常：红色。
6. 系统错误或硬件占用：红色闪烁或网页高优先级报警。

## 六、避障与禁区逻辑

### 6.1 障碍处理

遇到障碍物时，不应立即绕行。当前要求：

1. 超声波检测到前方障碍。
2. 小车停止。
3. 等待 6 秒。
4. 6 秒后再次读取超声波。
5. 如果障碍已消失，恢复原路线继续前进。
6. 如果障碍仍在，执行绕行避让。

### 6.2 绕行动作

以右侧绕行为例，动作序列为：

1. 原地顺时针旋转 90 度。
2. 向前行驶一个车身距离。
3. 逆时针回转 90 度。
4. 向前行驶一个车身距离。
5. 再次逆时针回转 90 度。
6. 向前行驶一个车身距离。
7. 顺时针转回 90 度，恢复原方向。

这个动作的目的不是自由导航，而是在固定通道中绕开障碍，同时最终恢复正向行驶。

### 6.3 嵌套避障

避障过程必须支持打断。

如果在绕行过程中又遇到新障碍：

1. 立即停止当前动作。
2. 进入新的障碍等待。
3. 等待 6 秒。
4. 若仍有障碍，嵌套执行新一轮绕行。
5. 新一轮绕行结束后，恢复上一层绕行流程。
6. 需要有最大嵌套次数或安全超时，避免无限递归导致小车反复绕行无法停止。

### 6.4 禁区处理

遇到“禁区”时，处理逻辑与遇到障碍物同理。

这里需要区分：

1. **列端四路全黑触发：** 属于正常端点转向逻辑，执行顺时针 90 度转弯。
2. **非预期禁区或绕行中压到禁区：** 按障碍处理，停止、等待、绕行或报警。

如果旧代码或旧文档把所有黑胶带都统一当成禁区停车，这会和真实需求冲突。实际实现必须先判断是不是“列端四路全黑触发”。

## 七、网页看板需求

### 7.1 当前网页问题

目前网页存在以下问题：

1. 网页显示了一个不明来源的“当前地图”网格。
2. 真实系统初始并没有预先固定地图，不应默认展示旧栅格地图。
3. 当前逻辑应是在检测到货架、物品、边界和禁区后，由软件逐步生成可视化地图或巡检轨迹。
4. 手动控制不可用，点击前进、转向、云台等功能时提示必须在 robot mode 中操作。
5. 用户在网页点“开始”后机器人没有启动，完全不会动。
6. 网页上的模拟演示能力和真实控制能力边界不清，容易让操作者误以为点击按钮就会控制真车。

### 7.2 地图显示要求

网页不应再默认展示一个固定栅格地图作为真实地图。

新的地图/可视化要求：

1. 初始状态可以显示“尚未生成巡检地图”或空白巡检轨迹。
2. 识别到货架后，在可视化中加入货架节点。
3. 识别到边界或列端后，在可视化中加入边界/转向点。
4. 识别到缺货或异常后，在对应货架节点上显示异常标记。
5. 可视化应表达“巡检轨迹和已识别货架”，而不是假装拥有完整 SLAM 地图。
6. 软件生成的地图可以是拓扑图、路线图、时间线或货架列示意图，不要求是真实比例二维地图。

### 7.3 控制按钮要求

网页必须提供真实可用的控制入口：

1. 开始巡逻。
2. 急停。
3. 停止/暂停。
4. 前进。
5. 后退。
6. 左移。
7. 右移。
8. 左转 90 度。
9. 右转 90 度。
10. 云台初始化。
11. 音频测试或预留播报。

要求：

1. 如果网页连接的是小车端服务，应允许真实控制。
2. 如果当前不是 robot mode，页面应明确显示“当前为模拟模式”，而不是只在点击后弹错误。
3. 手动控制不可用时，要告诉用户如何切换到 robot mode。
4. 点击“开始”必须能真正启动小车 runtime。
5. 所有危险动作都应有急停兜底。

### 7.4 状态显示要求

网页至少显示：

1. 当前运行模式：simulate 或 robot。
2. 当前是否连接真实硬件。
3. 小车状态：待命、巡逻中、扫描中、障碍等待、绕行中、禁区处理、异常报警、停止。
4. 当前货架。
5. 最近识别到的货架 AprilTag。
6. 最近识别到的物品。
7. 当前轮次。
8. 第一轮是否处于跳过缺货检测状态。
9. 障碍物距离。
10. 音频/灯光告警状态。
11. 最近事件列表。
12. 待确认异常列表。

## 八、系统运维需求

### 8.1 禁用官方 App 主程序

必须彻底禁用原来的 Yahboom App 主程序，避免占用摄像头、底盘、I2C 或端口资源。

约束：

1. 只能禁用官方 App 自启或杀掉官方 App 进程。
2. 不能破坏 Raspberry Pi 系统。
3. 不能删除系统级关键服务。
4. 不能影响 Wi-Fi、SSH、VNC、蓝牙音频等基础功能。
5. 操作前后应保留可恢复路径，例如记录被禁用的 autostart 文件路径。

### 8.2 我们的程序自启

目标：

1. 小车开机后自动启动我们的网页服务。
2. 小车开机后可以自动进入 robot runtime，或至少在网页点击开始后能立刻启动。
3. 自启服务应在启动前释放官方程序占用。
4. 程序异常退出后可以自动重启。
5. 日志应能通过 systemd 或文件查看。

### 8.3 网络要求

开机后必须保证：

1. Raspberry Pi 能正常连接 Wi-Fi。
2. 笔记本能通过小车 IP 访问网页看板。
3. SSH 能访问。
4. VNC 能访问。
5. 网页端口不被官方程序或旧进程占用。
6. 如果小车 IP 变化，应有清楚方法查询或配置。

### 8.4 访问方式

必须保留三种操作路径：

1. 网页看板：主要操作方式。
2. SSH：部署、查看日志、重启服务、诊断网络。
3. VNC：必要时进行图形界面操作和现场调试。

## 九、当前已知问题清单

### 9.1 需求与旧文档冲突

1. 旧文档仍大量描述固定栅格地图、A* 路径规划、避开黑胶带禁区。
2. 真实需求是货架通道循环巡逻，列端黑胶带触发转向。
3. 旧文档写任意一路压黑就停车后退，这与四路全黑端点触发逻辑冲突。
4. 旧配置只有 `A1/A2/B1/B2`，真实 A 列至少有 `A1/A2/A3/A4`。
5. 旧文档没有第一轮不检测缺货的要求。
6. 旧文档没有完整描述音频提示、语音警报和颜色可选识别。
7. 旧文档没有完整描述 6 秒等待、绕行动作和嵌套避障。
8. 旧网页计划默认显示固定地图，这与“运行后生成地图/轨迹”的要求冲突。

### 9.2 当前网页问题

1. 不明网格地图误导操作者。
2. 手动控制按钮不能直接控制真车。
3. 云台控制不能直接使用。
4. 点击开始不能让机器人运动。
5. simulate mode 和 robot mode 对用户不够透明。
6. 没有清楚显示当前是否连接硬件。

### 9.3 当前真车运行问题

1. 小车开机后未自动运动。
2. 官方 App 主程序可能仍然占用资源。
3. 我们的程序可能没有以 robot mode 启动。
4. 底盘运动、云台、传感器和网页 runtime 之间还没有完成可靠联调。
5. 手动控制入口没有成为现场调试的可靠工具。

### 9.4 当前配置问题

1. 货架数量与真实场地不一致。
2. 货架清单需要扩展到至少 A1-A4。
3. 物品识别配置需要表达图形、文字、AprilTag、颜色可选字段。
4. 缺少运行中生成地图/轨迹的数据结构。
5. 缺少巡检轮次、第一轮跳过检测、第二轮开始判断的明确状态字段。

## 十、验收标准

### 10.1 最小真车验收

1. 小车开机后 Wi-Fi 正常连接。
2. 网页看板可访问。
3. SSH 可访问。
4. VNC 可访问。
5. 官方 App 主程序不再占用底盘、摄像头或 I2C。
6. 网页显示当前为 robot mode。
7. 点击云台初始化，摄像头转向侧面。
8. 点击前进，小车低速前进。
9. 点击停止或急停，小车立即停止。
10. 点击开始巡逻，小车开始慢速巡逻。
11. 四路传感器同时检测到横向黑胶带时，小车顺时针原地转 90 度并继续。
12. 第一轮不报缺货。
13. 第二轮开始能在缺货时上报事件。

### 10.2 识别验收

1. 能识别货架 AprilTag。
2. 能识别至少一个物品 AprilTag。
3. 能通过文字识别至少一个物品或标签文本。
4. 能通过图形识别至少一个物品类别。
5. 有颜色时能记录颜色。
6. 没有颜色时不报错。
7. 识别到货架播放货架提示音。
8. 识别到物品播放物品提示音。
9. 检测到缺货时红灯提示并语音报警。

### 10.3 避障验收

1. 前方放置障碍后，小车停止。
2. 等待 6 秒。
3. 6 秒内移走障碍，小车恢复巡逻。
4. 6 秒后障碍仍在，小车执行绕行。
5. 绕行结束后恢复原方向继续前进。
6. 绕行过程中出现新障碍时，可以中断并嵌套执行新的避障逻辑。
7. 非预期禁区按障碍逻辑处理。

### 10.4 网页验收

1. 初始不显示假的固定地图。
2. 运行后根据识别到的货架、边界和事件逐步生成巡检地图或拓扑图。
3. 页面清楚显示 simulate/robot mode。
4. 页面清楚显示硬件连接状态。
5. 手动控制按钮可用于现场调试。
6. 开始按钮可以真正启动机器人。
7. 急停按钮始终可用。
8. 事件列表能显示缺货、错放、重复、未知、证据冲突。
9. 待确认异常可人工确认。
