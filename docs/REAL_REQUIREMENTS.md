# 真实需求基准文档

**状态：** 当前最重要的需求文档。  
**用途：** 记录真实小车场地、真实运行逻辑、网页控制、运维要求和当前问题。  
**优先级：** 如果本文与 `PROJECT_PLAN.md`、`docs/api_contract.md`、README 或当前代码配置冲突，以当前代码配置为准，并同步更新文档。

## 一、项目真实定位

本项目不是循线小车，也不是让小车沿黑线行驶。真实目标是：

**在固定货架通道中，让 RASPBOT V2 麦克纳姆轮小车自动往返巡逻，侧向扫描货架与物品，发现缺货等异常后用灯光、声音、网页看板进行提示和上报。**

小车的基本运行流程是固定的循环往复过程：

1. **初始位置与A列巡检：** 小车从初始位置启动，沿着 A 列货架进行慢速巡逻，侧向摄像头对准货架进行扫描。
2. **A列尽头与转弯：** 到达 A 列尽头时，检测到尽头禁区（横向黑胶带触发），小车停车并顺时针原地旋转 90 度。
3. **A端到B端过渡：** 旋转后，小车向 B 列通道对齐行驶，直至到达 B 端另一侧的尽头禁区。
4. **B端入口与转弯：** 到达 B 端尽头禁区后，小车再次顺时针原地旋转 90 度，对齐 B 列通道。
5. **B列巡检与避障：** 小车开始在 B 列货架进行慢速巡逻（此时摄像头对准 B 列货架，扫描顺序为 B3 到 B1）。在行驶过程中，B 列通道中部设有一处禁区（障碍），小车检测到后需执行绕行避让流程，绕过禁区后恢复方向，继续沿着 B 列货架行驶。
6. **B列尽头与转弯：** 行驶到 B 列末端的尽头禁区后，小车停车并顺时针原地旋转 90 度。
7. **B端到A端过渡：** 旋转后，小车返回 A 列通道一端，直至到达 A 端侧的尽头禁区。
8. **回到初始状态：** 到达尽头禁区后，小车再次顺时针原地旋转 90 度，回到最初在 A 列起点的朝向和状态。
9. **循环往复：** 整个流程不断循环进行。第一轮只运行建立状态不上报缺货；从第二轮开始实时检查并上报缺货等异常。

项目边界仍然保持清楚：

1. 不做 SLAM。
2. 不做未知环境自由自动驾驶。
3. 不让 LLM 控制底盘。
4. 不做机械臂搬运。
5. 不把黑胶带当作主行驶路线。
6. 地图不是预先人工画死的栅格路径；网页显示的地图应来自运行中检测到的货架、物品、边界和异常。

## 二、场地与货架布局

### 2.1 真实货架排列

当前已明确：

1. A 列货架按行驶方向依次为 `A1`、`A2`、`A3`、`A4`。
2. B 列货架为 `B1`、`B2`、`B3`。
3. 小车从 A 列转到 B 列边上之后，是从 B 列末端开始巡检，顺序为 `B3`、`B2`、`B1`。
4. 每个货架点都有一堆货物。
5. 小车沿货架列之间通道前进，侧向摄像头对准货架。
6. 到达 A 列或 B 列货架尽头时，小车会遇到横向黑胶带。
7. 横向黑胶带不是巡线轨道，而是列端/禁区触发标记。
8. 每列货架尽头的垂直方向安装“工字形”黑胶带，用于让小车在列端转弯，并在另一头再次转弯。
9. 通过这种方式，小车可以从 A 列切换到 B 列，摄像头转而对准 B 列货架，并从 B3 往 B1 方向巡检。

仍需确认：

1. 后续是否还有 C 列、D 列。
2. 每一列的货架编号方向是否都按小车实际经过顺序编号，还是需要在配置中单独记录实际经过顺序。
3. 每个货架的物品清单是否固定，还是允许现场调整后重新配置。

### 2.2 黑胶带语义

黑胶带有两类语义，必须区分：

1. **任意一路压黑：** 真实车默认立即锁存为边界命中，因为传感器只有压到黑胶带后才会读到黑色。
2. **列端或禁区语义：** 后续动作由当前巡逻阶段和最近货架锚点决定；列端执行转向，矩形禁区执行绕行。

端点动作的当前要求：

1. 停车。
2. 顺时针原地旋转 90 度。
3. 继续慢速前进。

注意：旧文档中“四路同时检测到黑胶带才触发”的描述已不符合当前上车调试。真实主逻辑是：任意一路压黑先停车锁存，再按当前阶段执行列端转向或禁区绕行，避免小车高速跨过边界。

## 三、巡逻运行逻辑

### 3.1 启动行为

小车开机时由人工放在初始位置。启动后应满足：

1. 云台先初始化，使摄像头朝向机身侧面货架。
2. 小车进入待命或自动巡逻状态。
3. 如果设置为开机自启，系统应在不需要人工 SSH 操作的情况下启动网页服务和机器人 runtime。
4. 如果使用网页启动，则点击“开始”后小车应立即慢速向前行驶。
5. 开始后网页应明确显示当前状态为巡逻中、移动中或扫描中。

当前遇到的问题：

1. 点击网页“开始”后机器人没有动。
2. 之前小车开机半天不动，说明 runtime 没有真正接管底盘，或没有以正确模式启动。
3. 手动控制按钮提示必须进入 robot mode，导致前进、转向、云台等功能无法直接使用。

### 3.2 循环检测

巡逻必须是持续循环，而不是跑完一条固定路径就结束。

循环逻辑：

1. 第 1 轮：只运行、校准、识别经过的货架，不上报缺货。
2. 第 2 轮及之后：开始按货架清单检查物品，发现缺失、重复、错放、未知物品或证据冲突时实时上报。
3. 每次经过货架时，如果识别到货架 AprilTag，更新当前货架。
4. 每次识别到物品时，记录该物品属于当前货架的一次观察。
5. 每轮结束时，系统可对这一轮看到的物品集合与预期清单进行比对。
6. 如果需要实时性，也可以在离开某个货架区域后立即对该货架做局部比对。

### 3.3 行驶规范

小车应尽量保持行驶方向为正向，也就是通过绕行和转向动作恢复原行驶方向，而不是长期倒车或以错误朝向巡逻。

原则：

1. 正常巡逻时低速前进。
2. 到货架尽头时按端点逻辑转向。
3. 遇到障碍或禁区需要绕行时，优先向外侧绕行，避免向货架方向挤压。
4. 绕行结束后恢复原方向，继续向前巡逻。
5. 不应因为临时避障导致摄像头长期背向货架或行驶方向混乱。

## 四、识别需求

### 4.1 货架识别

货架通过 AprilTag 识别。

要求：

1. 每个货架必须贴有货架 AprilTag。
2. AprilTag 是货架身份的主来源。
3. 货架上的大号文字，如 `A1`、`A2`，可以作为 OCR 辅助校验，但不能替代 AprilTag 主身份。
4. 识别到货架后，系统应更新当前货架编号。
5. 识别到货架时，应播放货架扫描提示音。

### 4.2 物品识别

物品识别需要支持三种形式：

1. 图形识别。
2. 文字识别。
3. AprilTag 识别。

其中 AprilTag 可以作为稳定身份来源；图形和文字可以作为补充证据。后续如果实际物品没有 AprilTag，也应允许通过图形和文字形成可用观察，但置信度和异常判断策略要更保守。

颜色识别要求：

1. 如果物品有颜色标识，应识别颜色。
2. 支持至少蓝、绿、黄、红等常见颜色。
3. 颜色用于物品价值、属性或复核判断。
4. 如果没有颜色，系统不能报错。
5. 缺少颜色时，应将颜色字段记为未知、空值或未提供，而不是当作异常。

识别到具体物品时，应播放另一个物品提示音，与货架扫描提示音区分。

### 4.3 异常判断

从第二轮巡逻开始，系统需要判断：

1. `missing_item`：预期存在但当前轮未识别到。
2. `duplicate_item`：同一货架上识别到重复物品。
3. `wrong_shelf`：物品出现在错误货架。
4. `unknown_item`：识别到不在配置表中的物品。
5. `evidence_mismatch`：AprilTag、图形、文字、颜色之间存在冲突。

缺货异常的现场反馈：

1. 亮红色 LED。
2. 发出语音警报。
3. 语音内容应能表达类似“检测到某某物品缺失”。
4. 网页看板显示缺失物品、所在货架、时间、事件状态。
5. 事件进入待确认或待处理状态。

### 4.4 算法与硬件选型建议

推荐采用一套 **“传感器触发 + 视觉确认 + OCR 读取 + 语音反馈”** 的分层方案，不把所有判断都压在一个大视觉模型或大语言模型上。

推荐硬件组合：

1. Raspberry Pi 5 8GB。
2. AI HAT+ 13 TOPS。
3. Camera Module 3 标准视角版。
4. MPU6050 加速度/陀螺仪模块。
5. 主动散热。

选型理由：

1. 树莓派 5 负责业务状态机、AprilTag、OCR 调度、GPIO、网页和日志。
2. AI HAT+ 负责轻量目标检测，把神经网络推理从 CPU 上卸下来。
3. 13 TOPS 对单相机、少类别、按需 OCR 的货架巡检已经足够。
4. 26 TOPS 更适合未来多路相机、更大模型或更复杂视觉任务，不是当前最小演示必需。
5. Camera Module 3 有自动对焦，适合近距离读取卡片文字；如果卡片距离固定，后续可以锁定焦点，减少自动对焦带来的延迟。
6. MPU6050 通过 I2C 接入，用于读取加速度和陀螺仪数据，辅助判断小车姿态变化、碰撞/剧烈晃动和 90 度转向标定；它不是主导航来源，不替代黑胶带列端触发、超声波避障或视觉识别。

各能力分工：

1. **传感器：** 超声波、红外和四路黑胶带传感器负责触发流程，例如发现障碍、进入扫描区、到达列端。
2. **MPU6050：** 提供加速度和角速度采样，辅助运动状态判断、转向角度标定和异常晃动记录。
3. **AprilTag：** 负责货架或卡片身份、位置、角度、四角位置、是否稳定、是否处理过。
4. **轻量目标检测：** 只判断画面里是否有卡片、货架槽位或未知物体，以及目标框是否进入扫描 ROI。
5. **OpenCV：** 负责卡片轮廓、透视矫正、ROI 裁剪和必要的颜色判断。
6. **OCR：** 只在卡片稳定后，对裁剪和透视矫正后的局部文字区域进行识别。
7. **业务状态机：** 判断当前目标、已完成目标、下一目标、轮次、缺货和异常。
8. **Piper 或固定音频：** 负责离线语音播报。

第一阶段建议不训练模型也能完成演示：

1. AprilTag 定位。
2. 卡片矩形轮廓。
3. OpenCV 透视矫正。
4. OCR 读取。
5. 规则校验和语音反馈。

第二阶段再加入自训练轻量检测模型：

1. 训练少数类别，例如 `card`、`shelf_slot`、`unknown_object`。
2. 使用 YOLOv8n 或同等级轻量模型。
3. 推理放到 AI HAT+ 上。
4. 低频持续检测即可，例如 5-10 FPS，不需要 30 FPS。
5. 目标检测只负责“是不是卡片、在哪里、是否进入扫描框”，不负责理解卡片内容。

不要一开始做“识别所有物体是什么”的通用模型。课程项目里，类别越少、规则越清楚，现场成功率越高。

推荐视觉状态机：

```text
IDLE
  -> 传感器发现目标靠近
SEARCHING
  -> 低频目标检测 + AprilTag
ALIGNING
  -> 目标进入 ROI，角度和连续帧稳定
SLOW_DOWN_OR_STOP
  -> 减速或暂停
CAPTURE
  -> 拍高分辨率清晰图
RECTIFY
  -> 按 AprilTag 或卡片四角透视矫正
OCR
  -> 只识别局部文字区域
VERIFY
  -> 置信度、编号格式、预置库匹配
TTS
  -> 播报识别结果或异常
DONE
  -> 记录已处理目标，等待离开 ROI
SEARCHING
  -> 检测下一目标
```

关于“当前物体 / 已检测物体 / 下一个物体”的判断：

1. 当前对准物体：目标检测框中心落在当前扫描 ROI，且 AprilTag 位置正确。
2. 已检测物体：检测模型连续多帧识别为 `card` 或目标物，且 OCR/AprilTag 已完成处理。
3. 下一个物体：当前目标完成并离开 ROI 后，新目标轨迹进入 ROI。

“下一个物体”不依赖模型理解语义，而由状态机判断“当前完成并离开、新目标进入”。

OCR 推荐：

1. 使用 PaddleOCR 轻量中文模型。
2. 只对裁剪、透视矫正后的卡片局部区域做 OCR。
3. 不使用复杂文档解析或视觉语言模型作为主方案。
4. 加三层校验：置信度阈值、编号格式校验、预置货架/物品库比对。
5. 识别不可靠时提示“请保持卡片稳定”，不要硬猜。

语音推荐：

1. 使用 Piper 中文离线 TTS，或第一阶段先用固定音频文件。
2. 只在 OCR 或异常判断完成后播报短句。
3. 语音播放使用队列，避免连续识别时互相打断。
4. 语音失败不能影响巡逻和识别主流程。

推荐软件分工：

| 模块 | 建议技术 |
|---|---|
| 摄像头采集 | `libcamera` / Picamera2 |
| 卡片/目标检测 | YOLO 轻量模型，运行在 AI HAT+ |
| 定位与姿态 | AprilTag 3 + OpenCV |
| 卡片边框与透视矫正 | OpenCV |
| 中文 OCR | PaddleOCR 轻量模型 |
| 离线语音 | Piper，或第一阶段固定音频 |
| 超声波 / 红外 / 黑胶带 | GPIO/I2C 低频轮询或中断 |
| 整体调度 | Python 状态机，必要时线程或 asyncio |
| 结果记录 | SQLite 或 JSON 文件 |

## 五、音频与灯光提示

### 5.1 音频提示类型

至少需要三类音频/语音能力：

1. **货架提示音：** 巡检过程中扫描到货架时播放。
2. **物品提示音：** 识别到具体物品时播放。
3. **异常语音警报：** 识别到缺货、错放等需要处理的异常时播放。

当前已准备的固定音效：

| Cue | 触发场景 | 项目文件 | 部署后静态路径 |
|---|---|---|---|
| `obstacle` | 检测到障碍物、非预期禁区或需要进入障碍/禁区处理流程时播放 | `src/inspection_robot/static/audio/obstacle.wav` | `/static/audio/obstacle.wav` |
| `first` | 检测到货架、确认进入货架扫描流程时播放 | `src/inspection_robot/static/audio/first.wav` | `/static/audio/first.wav` |
| `following` | 检测到货架上的具体物品时播放；每检测到一个物品播放一次 | `src/inspection_robot/static/audio/following.wav` | `/static/audio/following.wav` |

这些音效来自 `/Users/kmg/Desktop/sounds/*.MP4`，已转换为树莓派播放更稳的 `PCM 16-bit WAV`、`44.1 kHz`、单声道格式。后续如果替换源文件，仍应保持同名 cue 和同一路径，避免业务逻辑频繁调整。

### 5.2 异常语音

缺货语音示例：

```text
检测到 A2 货架 item_04 缺失，请及时补货。
```

语音系统第一阶段可以只预留接口，第二阶段接入固定音频文件，第三阶段再做动态 TTS。

要求：

1. 没有音频文件时，不应影响小车巡逻。
2. 音频播放失败时，应记录事件或状态，但不能让主程序崩溃。
3. 网页应提供测试播放或预留播报按钮。
4. 小车本体音频输出优先通过树莓派连接的蓝牙音箱或本地音频设备播放，而不是浏览器所在电脑播放。
5. 音频播放必须非阻塞，不得卡住巡逻、避障、扫描或网页请求。
6. 播放应通过树莓派默认音频设备输出，优先使用系统可用播放器如 `paplay`、`pw-play`、`aplay` 或 `ffplay`。
7. 连续触发时应使用队列或节流策略，避免多个音效互相堆叠导致听不清。

### 5.3 灯光提示

建议状态：

1. 正常巡逻：绿色或低亮度常规状态。
2. 扫描货架：绿色短闪。
3. 识别物品：蓝色或白色短闪。
4. 障碍等待：蓝色或紫色。
5. 缺货、错放、高优先级异常：红色。
6. 系统错误或硬件占用：红色闪烁或网页高优先级报警。

## 六、避障与禁区逻辑

### 6.1 障碍处理

遇到障碍物时，不应立即绕行。当前要求：

1. 超声波检测到前方障碍。
2. 小车停止。
3. 等待 6 秒。
4. 6 秒后再次读取超声波。
5. 如果障碍已消失，恢复原路线继续前进。
6. 如果障碍仍在，执行绕行避让。

### 6.2 绕行动作

以右侧绕行为例，动作序列为：

1. 原地顺时针旋转 90 度。
2. 向前行驶一个车身距离。
3. 逆时针回转 90 度。
4. 向前行驶一个车身距离。
5. 再次逆时针回转 90 度。
6. 向前行驶一个车身距离。
7. 顺时针转回 90 度，恢复原方向。

这个动作的目的不是自由导航，而是在固定通道中绕开障碍，同时最终恢复正向行驶。

### 6.3 嵌套避障

避障过程必须支持打断。

如果在绕行过程中又遇到新障碍：

1. 立即停止当前动作。
2. 进入新的障碍等待。
3. 等待 6 秒。
4. 若仍有障碍，嵌套执行新一轮绕行。
5. 新一轮绕行结束后，恢复上一层绕行流程。
6. 需要有最大嵌套次数或安全超时，避免无限递归导致小车反复绕行无法停止。

### 6.4 禁区处理

遇到“禁区”时，处理逻辑与遇到障碍物同理。

这里需要区分：

1. **列端边界触发：** 任意一路压黑后先停车锁存，再按当前阶段执行顺时针 90 度转弯。
2. **非预期禁区或绕行中压到禁区：** 任意一路压黑后先停车锁存，再按障碍处理、绕行或报警。

如果旧代码或旧文档要求四路全黑才触发，这会和当前真实车传感器特性冲突。实际实现必须先锁存压黑瞬间，再按当前巡逻阶段判断列端或禁区动作。

## 七、网页看板需求

### 7.1 当前网页问题

目前网页存在以下问题：

1. 网页显示了一个不明来源的“当前地图”网格。
2. 真实系统初始并没有预先固定地图，不应默认展示旧栅格地图。
3. 当前逻辑应是在检测到货架、物品、边界和禁区后，由软件逐步生成可视化地图或巡检轨迹。
4. 手动控制不可用，点击前进、转向、云台等功能时提示必须在 robot mode 中操作。
5. 用户在网页点“开始”后机器人没有启动，完全不会动。
6. 网页上的模拟演示能力和真实控制能力边界不清，容易让操作者误以为点击按钮就会控制真车。

### 7.2 地图显示要求

网页不应再默认展示一个固定栅格地图作为真实地图。

新的地图/可视化要求：

1. 初始状态可以显示“尚未生成巡检地图”或空白巡检轨迹。
2. 识别到货架后，在可视化中加入货架节点。
3. 识别到边界或列端后，在可视化中加入边界/转向点。
4. 识别到缺货或异常后，在对应货架节点上显示异常标记。
5. 可视化应表达“巡检轨迹和已识别货架”，而不是假装拥有完整 SLAM 地图。
6. 软件生成的地图可以是拓扑图、路线图、时间线或货架列示意图，不要求是真实比例二维地图。

### 7.3 控制按钮要求

网页必须提供真实可用的控制入口：

1. 开始巡逻。
2. 急停。
3. 停止/暂停。
4. 前进。
5. 后退。
6. 左移。
7. 右移。
8. 左转 90 度。
9. 右转 90 度。
10. 云台初始化。
11. 音频测试或预留播报。

要求：

1. 如果网页连接的是小车端服务，应允许真实控制。
2. 如果当前不是 robot mode，页面应明确显示“当前为模拟模式”，而不是只在点击后弹错误。
3. 手动控制不可用时，要告诉用户如何切换到 robot mode。
4. 点击“开始”必须能真正启动小车 runtime。
5. 所有危险动作都应有急停兜底。

### 7.4 状态显示要求

网页至少显示：

1. 当前运行模式：simulate 或 robot。
2. 当前是否连接真实硬件。
3. 小车状态：待命、巡逻中、扫描中、障碍等待、绕行中、禁区处理、异常报警、停止。
4. 当前货架。
5. 最近识别到的货架 AprilTag。
6. 最近识别到的物品。
7. 当前轮次。
8. 第一轮是否处于跳过缺货检测状态。
9. 障碍物距离。
10. 音频/灯光告警状态。
11. 最近事件列表。
12. 待确认异常列表。

### 7.5 实时视频与检测结果展示要求

为了让操作者在现场调试和演示时能够直观地看到小车的感知状态，网页应当：

1. **实时画面预览：** 网页第一页（主界面）需从小车摄像头获取实时拍摄图像，并在主面板进行嵌入。
2. **分辨率与流畅度：** 画面分辨率可以设置为较低分辨率（如 320x240 或 640x480）以减少传输带宽和 CPU 占用，但必须保证视频流传输的流畅度（即高且稳定的帧率），确保无卡顿。
3. **检测结果叠加与展示：** 画面上或其侧边应动态直观地渲染出视觉算法的检测结果（如当前识别到的 AprilTag ID、检测到的物品名称、OCR 标签内容等各种多模态感知证据），以便调试比对。

## 八、系统运维需求

### 8.1 禁用官方 App 主程序

必须彻底禁用原来的 Yahboom App 主程序，避免占用摄像头、底盘、I2C 或端口资源。

约束：

1. 只能禁用官方 App 自启或杀掉官方 App 进程。
2. 不能破坏 Raspberry Pi 系统。
3. 不能删除系统级关键服务。
4. 不能影响 Wi-Fi、SSH、VNC、蓝牙音频等基础功能。
5. 操作前后应保留可恢复路径，例如记录被禁用的 autostart 文件路径。

### 8.2 我们的程序自启

目标：

1. 小车开机后自动启动我们的网页服务。
2. 小车开机后必须保持静止，禁止自动进入 robot runtime；只有网页点击开始巡逻后才能启动。
3. 自启服务应在启动前释放官方程序占用。
4. 程序异常退出后可以自动重启。
5. 日志应能通过 systemd 或文件查看。

### 8.3 网络要求

开机后必须保证：

1. Raspberry Pi 能正常连接 Wi-Fi。
2. 笔记本能通过小车 IP 访问网页看板。
3. SSH 能访问。
4. VNC 能访问。
5. 网页端口不被官方程序或旧进程占用。
6. 如果小车 IP 变化，应有清楚方法查询或配置。

### 8.4 访问方式

必须保留三种操作路径：

1. 网页看板：主要操作方式。
2. SSH：部署、查看日志、重启服务、诊断网络。
3. VNC：必要时进行图形界面操作和现场调试。

## 十、验收标准

### 10.1 最小真车验收

1. 小车开机后 Wi-Fi 正常连接。
2. 网页看板可访问。
3. SSH 可访问。
4. VNC 可访问。
5. 官方 App 主程序不再占用底盘、摄像头或 I2C。
6. 网页显示当前为 robot mode。
7. 点击云台初始化，摄像头转向侧面。
8. 点击前进，小车低速前进。
9. 点击停止或急停，小车立即停止。
10. 点击开始巡逻，小车开始慢速巡逻。
11. 任意一路传感器压到横向黑胶带时，小车立即锁存停车，并在列端阶段顺时针原地转 90 度后继续。
12. 第一轮不报缺货。
13. 第二轮开始能在缺货时上报事件。

### 10.2 识别验收

1. 能识别货架 AprilTag。
2. 能识别至少一个物品 AprilTag。
3. 能通过文字识别至少一个物品或标签文本。
4. 能通过图形识别至少一个物品类别。
5. 有颜色时能记录颜色。
6. 没有颜色时不报错。
7. 识别到货架播放货架提示音。
8. 识别到物品播放物品提示音。
9. 检测到缺货时红灯提示并语音报警。

### 10.3 避障验收

1. 前方放置障碍后，小车停止。
2. 等待 6 秒。
3. 6 秒内移走障碍，小车恢复巡逻。
4. 6 秒后障碍仍在，小车执行绕行。
5. 绕行结束后恢复原方向继续前进。
6. 绕行过程中出现新障碍时，可以中断并嵌套执行新的避障逻辑。
7. 非预期禁区按障碍逻辑处理。

### 10.4 网页验收

1. 初始不显示假的固定地图。
2. 运行后根据识别到的货架、边界和事件逐步生成巡检地图或拓扑图。
3. 页面清楚显示 simulate/robot mode。
4. 页面清楚显示硬件连接状态。
5. 手动控制按钮可用于现场调试。
6. 开始按钮可以真正启动机器人。
7. 急停按钮始终可用。
8. 事件列表能显示缺货、错放、重复、未知、证据冲突。
9. 待确认异常可人工确认。
10. 网页第一页包含流畅的实时摄像头视频预览，并在画面上或侧边直观地渲染显示当前的感知识别与检测结果。

## 十一、实现优先级

### P0：必须先解决
